Implementation Plan

App- & Role-level Session (Idle) Timeout Configuration

Zango · zelthy3 + pkg-zango-appbuilder · 2026-07-09 · Status: Draft · Mirrors PR #601 (auth token TTL)

Overview

Session idle timeout is currently platform-global. django-session-security is configured once via SESSION_SECURITY_WARN_AFTER (1700s) and SESSION_SECURITY_EXPIRE_AFTER (1800s) in config/settings/base.py. Every app (tenant) is forced to identical idle-warning and auto-logout timing.

Goal: make idle timeout (both the warning and the auto-logout) configurable per app (Auth Config) and overridable per user role, falling back to the platform default — reusing the exact model already shipped for auth token TTL in PR #601 (67147413).

Precedence: user_role > tenant > platform default — resolved through the existing get_auth_priority(policy="session_policy", ...) deep-merge.
Backend-only — zero template changes. Two client frontends exist (appbuilder React + legacy Django template). Both already receive their warn/expire numbers from the server, so every lever below is a backend change. No workspace _base.html is touched.

The workspace template only does {% include 'session_security/all.html' %}. That stock include prints its numbers via the template filters warn_after / expire_after (from session_security_tags), which today return the global constants. These filters receive request → the override point, with no workspace edit.

The enforcer · session_security middleware (both frontends)

SessionSecurityMiddleware.process_request() logs the user out (clears the session) when idle past get_expire_seconds(request) — which returns the module-level global constant. On the next ping, the cleared session makes PingView return "logout", and both clients call expire(). get_expire_seconds(request) is an override hook.

Key insight: the middleware is the single authoritative logout for both frontends (client JS can't bypass it). The warning modal, however, is a purely client-side timer — so per-app/role warning timing requires feeding the client the right numbers (Frontends A & B above).

Target Design

Store two new optional keys inside the existing session_policy block (seconds; absent = inherit platform default):

session_policy: {
    max_concurrent_sessions,     // existing
    token_ttl,                   // existing (#601)
    session_warn_after,          // NEW — seconds; absent = inherit
    session_expire_after,        // NEW — seconds; absent = inherit
}

Steps

  1. Schema — add keys
    In apps/shared/tenancy/schema.py, add session_warn_after: int and session_expire_after: int to SessionPolicy (already total=False). UserRoleAuthConfig already carries session_policy from #601 — no role change.
  2. Resolver — get_app_session_timeout()
    In core/utils.py, add next to get_app_token_ttl. Call get_auth_priority(policy="session_policy", ...), return (warn_after, expire_after), each falling back to settings.SESSION_SECURITY_WARN_AFTER / EXPIRE_AFTER. Guard: if expire <= warn or only one set, fall back to defaults.
  3. Custom middleware — server-side logout (both frontends)
    New middleware/session_security.py subclassing SessionSecurityMiddleware, overriding get_expire_seconds(request) to resolve via get_app_session_timeout using request.tenant + role. Swap the entry in settings/base.py (line 108). Only expire_after matters server-side; this is the authoritative logout for React and legacy apps (client JS can't bypass it).
  4. Serializer — validation + platform hints
    In api/platform/tenancy/v1/serializers.py, extend session_policy validation for the two new keys (int ≥ 0, warn < expire, sane upper bound). Inject read-only platform_session_warn_after / platform_session_expire_after in to_representation and strip on write — same pattern as platform_token_ttl. Apply to tenant + role serializers.
  5. Appbuilder (React) — serve per-app/role config
    In appbuilder/backend/app/utils.py::get_session_security_config(), accept user/user_role/tenant and call get_app_session_timeout(...) instead of reading raw settings. Update caller in views.py::AppInitializeAPI.get (already has all three). useSessionSecurity.ts already consumes warn/expire dynamically — no hook change; just confirm e2e mocks (frontend/e2e/mocks/app-initializer-*.json) still align.
  6. Legacy apps — override session_security templatetag filters
    Add templatetags/session_security_tags.py in a Zango app that loads before session_security in INSTALLED_APPS. Redefine the warn_after / expire_after filters to return per-request values from get_app_session_timeout(request). The stock all.html and every workspace _base.html stay unchanged — {{ request|warn_after }} simply resolves to the new filter. (Zango already shadows library tags, e.g. zstatic.)
  7. Auth Config UI — mirror TokenTtlField
    Add SessionTimeoutField.jsx (or extend the "Session & Tokens" step) with two humanized duration inputs + inherit/custom mode, reusing helpers from utils/tokenTtl.js. Wire into ModernAuthConfig.jsx (app) and RoleOverrideModal.jsx (role), with "App default" / "Platform default" inherit labels from the platform_session_* hints.

Dependencies & Migration

  • Reuses existing get_auth_priority deep-merge, Knox settings pattern, and tokenTtl.js humanize/unit helpers — no new packages.
  • No DB migration — new keys are optional entries in the existing auth_config JSONField; absence = inherit. Existing apps/roles keep 1700/1800.

Risk Notes

⚠ INSTALLED_APPS ordering (templatetag override): Django loads the first session_security_tags it finds — the Zango app providing the override must be listed before "session_security". If not, the override is silently ignored. Verify by inspecting the rendered warnAfter/expireAfter in a legacy app's page source.
⚠ Middleware role timing: the custom middleware runs before UserRoleAndAppObjectAssignmentMiddleware (base.py 108 vs 109), so role may not be attached yet for the logout path. Resolve role as get_auth_priority does internally, and use the tenant-level expire_after as a safe floor. (The templatetag filters and the React config builder run inside views, where role is fully resolved — so warnings are exact.)
⚠ warn ≥ expire misconfig: validate at the serializer and defensively in get_app_session_timeout — an inverted pair breaks the countdown UX.
⚠ Client-only bypass: frontend timers are advisory; the middleware subclass is what actually enforces logout. Verify both frontends.

Verification

  1. Resolution unit test — get_app_session_timeout with tenant-only, role-override, and neither set; assert precedence, platform fallback, and warn < expire guard.
  2. Serializer — PATCH tenant auth_config with valid + invalid values (negative, warn≥expire, non-int); assert 400s and that platform_session_* is never persisted.
  3. App initializer (React) — hit AppInitializeAPI as two roles in one app with different overrides; assert session_security_config warn/expire differ per role.
  4. Legacy template — load a non-appbuilder app (e.g. newmicares) as two roles; inspect page source and assert the rendered warnAfter/expireAfter reflect per-role config, with the workspace _base.html unmodified.
  5. Frontend timers — run frontend/e2e/session-security/* against a short-timeout mock; confirm warning + auto-logout fire at configured values.
  6. Server-side enforcement (both frontends) — short per-app expire_after, idle past it, then request; confirm the middleware logs out (302/401 → "logout" ping) for a React app and a legacy app, even with client JS disabled.
  7. Regression — an app/role with no session_policy override still uses 1700/1800.
