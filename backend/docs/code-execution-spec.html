01 · Overview

What we're building

Developers and platform admins need a controlled, auditable way to run ad-hoc Python inside a tenant's workspace — without SSHing into a worker, dropping into a Django shell, or scripting one-off celery tasks. Code Execution is a platform-managed catalog of named snippets with a tracked execution history. Every run goes through the same sandboxed code path as scheduled tasks.

Today, the only ways to run arbitrary code in a tenant context are tenant_command shell, manually written AppTask entries scheduled via celery beat, or one-off management commands. None of these are self-serve, auditable, or visible in the admin panel. This feature closes that gap with a designed UI and a hardened execution path.

02 · Outcomes

Goals & non-goals

Three primary goals, ordered by priority. Non-goals at the end keep scope honest.

▸

Self-serve, in-tenant execution

Authorized users create, edit, and run named snippets from the admin panel. Each run executes inside the workspace's plugin source, with full access to dynamic models — exactly like a scheduled AppTask.

◐

Auditable history

Every run is a row. Captured stdout, return value, exception traceback, duration, triggered-by, and a snapshot of the code that ran (so edits don't rewrite history).

⊘

Safe by construction

AST-level deny list blocks dangerous imports, attribute access, and dunder reflection at save and at run. Permission gated. Wall-clock timeout. Output capped.

⚠

Hard tenant isolation — partial (L2 parked)

A snippet running in tenant bmspspmonitoring should not be able to read or modify data in tntApp. The original design enforced this at four layers; L2 (Postgres role) is currently parked by product direction (no DB-level role changes at this time). The three active layers — AST (L1), locked search_path (L3), and ConnectionGuard (L4) — block the obvious paths, but cross-tenant access via fully-qualified raw SQL remains possible until L2 is restored. See § Tenant isolation.

Non-goals (v1): Scheduling, parameterization with form inputs, multi-file snippets, IDE-style linting, draft/PR workflow, cross-tenant snippets, debugger / breakpoints. These are deliberately deferred. v1 is one file, one run-button, full history.

Two file scopes — and how they connect

The editor modal manipulates CodeSnippetFile rows — they're saved with the snippet and shown every time the modal is opened. Each run snapshots the current snippet files into CodeExecFile(kind=input) rows, so editing the snippet's files later cannot rewrite the history of past runs.

Save
Files uploaded in the modal are persisted as CodeSnippetFile rows. No CodeExecution row is created.
Save & Run · Run
Persist CodeSnippetFile as above (if Save & Run), then snapshot every snippet file into CodeExecFile(kind=input) tied to the new CodeExecution.
During run
codexec.write(...) inserts CodeExecFile(kind=output). Outputs always live at the execution level only — they belong to a specific run.

Cheap snapshot: the bytes aren't copied. CodeExecFile.storage_key points at the same blob as CodeSnippetFile.storage_key for snapshotted inputs, and source_snippet_file records the provenance. Storage objects are reference-counted at GC time — a snippet file is only deleted once no execution still references it.

Why snapshot source on each run? If the snippet is edited after a run completes, the history must still reflect what actually executed. Hash is stored alongside for cheap deduplication and tamper detection. Loose coupling via FK preserves the history when a snippet is archived.

✓ Every run history row carries

Code at run
Full source_snapshot + source_hash. Edits after the run don't rewrite history.
Logs of run
stdout + stderr on the row · per-line tail in CodeExecutionLogLine.
Files in & out
Every uploaded input AND every emitted output as a CodeExecFile row (FK to this run, kind=input/output). Downloadable until 90-day GC.
Result & trace
return_value, exception_type, exception_traceback, duration, who triggered.

A run history row is fully self-contained — you can re-derive what code executed, what data went in, what came out, and what was logged, even after the snippet is edited or archived.

07 · Security

Deny list & validation

We are not sandboxing the Python runtime. We are restricting at parse time via AST traversal, and at runtime via process limits. Anyone with the policy can do anything the worker can do — this is a privileged feature gated by permission, not a sandbox.

Runtime guarantees

  • Wall-clock timeout_seconds per snippet (default 60, max 300). Enforced via celery's soft_time_limit.
  • stdout capped at 1 MB per run · further writes silently dropped with a final SYS line.
  • stderr capped at 256 KB. Tracebacks always preserved.
  • Single line of stdout capped at 8 KB. Truncated with … marker.
  • One concurrent run per snippet · second invocation returns 409.
  • Tenant-isolated celery queue · job carries tenant name & user id.
  • Execution runs as SystemUser · all auditlog rows tag the triggering user.
  • Aborting flips status to aborted & revokes celery task with terminate=True.
  • Worker pool isolation: dedicated codexec queue · worker concurrency low (≤4).
  • AST check runs twice · at save and at job pickup · catches deny-list updates mid-flight.
Permissions: Three scoped policies — platform.code_execution.use (view + list), .edit (create/update/archive snippets), and .run (trigger an execution). Tenants opt in per role. Default: only Platform Admin and Developer roles get .run.
07b · Tenant isolation

Cross-tenant query barrier

The single most important invariant: a snippet running in tenant A cannot read or write tenant B's schema — by ORM, by raw SQL, or by any creative reflection. The public schema (shared registry, packages, periodic-task tables) remains readable. Enforced as four overlapping layers, so a bypass at one layer hits a wall at the next.

✓

Allowed from tenant A

ORM & raw SQL against tenant_a_schema.*
Read-only SQL against public.* shared tables
Django shared models (tenant registry list, package metadata)

✕

Blocked from tenant A

Any ORM query that resolves to tenant_b_schema
Raw SQL referencing another tenant's schema name
connection.set_tenant(B) / schema_context("B")

⚠

Blocked everywhere

Writes to public beyond a narrow audit-only allowlist
SET search_path from user code
Direct execution against django.db.connections["default"] bypassing the guard

⚠
Status: L2 parked. Per product direction we are not currently provisioning per-tenant Postgres roles. The three active layers below (L1 AST, L3 search_path, L4 ConnectionGuard) block the common-case paths and reflective bypasses. They do not prevent a snippet from running fully-qualified raw SQL like SELECT … FROM "tntApp"."x", which the app's current DB user can execute. To close this gap, restore L2: provision a codexec_<schema> role per tenant and add SET LOCAL ROLE back to the executor's transaction.

Four-layer defense

L1

AST parse-time block

Developer-facing layer. Stops obvious tenant-switching attempts at save with a precise error pointing at the offending line.

Import of django_tenants.utils.* connection.set_tenant(...) connection.set_schema(...) connection.schema_name = ... connection.tenant = ... schema_context(...) tenant_context(...) TenantModel.objects.* anywhere outside list_tenants() string literal matches /SET\s+search_path/i string literal matches /SET\s+SCHEMA/i
L2

Postgres role with schema-scoped grants parked — not implemented

PARKED. The original design — and still the correct cross-tenant defense — uses a per-tenant codexec_<schema> role with USAGE only on its own schema + read-only on public. Code execution would SET LOCAL ROLE to this role at run time so cross-schema queries fail at the DB with permission denied. Provisioning Postgres roles is a deployment-level change we're not making at this time. The code below shows what would be applied if we restore it.

1234567891011
-- Provisioned per tenant at schema-create timeCREATE ROLE codexec_bmspspmonitoring NOLOGIN;GRANT codexec_bmspspmonitoring TO zango_app_role; -- so app can SET ROLEREVOKE ALL ON ALL TABLES IN SCHEMA "tntApp" FROM codexec_bmspspmonitoring;REVOKE USAGE ON SCHEMA "tntApp" FROM codexec_bmspspmonitoring;GRANT USAGE ON SCHEMA bmspspmonitoring TO codexec_bmspspmonitoring;GRANT ALL ON ALL TABLES IN SCHEMA bmspspmonitoring TO codexec_bmspspmonitoring;GRANT USAGE ON SCHEMA public TO codexec_bmspspmonitoring;GRANT SELECT ON ALL TABLES IN SCHEMA public TO codexec_bmspspmonitoring;

Result: SELECT * FROM "tntApp"."patient" from inside this tenant's code execution returns "permission denied for schema tntApp" at the SQL layer — no application logic involved.

L3

Locked search_path + statement_timeout active · partial defense

Before exec(), the worker opens a transaction and pins the search_path. Any user-issued SET search_path is rolled back when the transaction ends. Unqualified table references like FROM client always resolve to this tenant's client — never another tenant's. Also caps any single query at the snippet's wall-clock budget via statement_timeout.

1234567
with transaction.atomic(): with connection.cursor() as c: c.execute('SET LOCAL search_path = "bmspspmonitoring", public') c.execute('SET LOCAL statement_timeout = 60000') # user code runs here exec(compile(source_snapshot, '<codexec>', 'exec'), namespace)# transaction ends — search_path + timeout revert automatically

Limitation without L2: a snippet using fully-qualified raw SQL (e.g., SELECT * FROM "tntApp"."patient") bypasses search_path entirely. L1 catches the obvious SQL fragments in string literals; for full coverage we need L2 restored.

L4

Python connection guard

Final belt to the suspenders. The DatabaseWrapper exposed to user code is wrapped in a proxy that raises PermissionError("tenant switching is not permitted in code execution") when set_tenant, set_schema, or assignment to schema_name / tenant is attempted. Catches obfuscated bypasses that slipped past the AST (e.g., getattr(connection, "set_" + "tenant")(...)).

i
The guard is also what flushes captured stdout/stderr — it sits in the same wrapping layer, so disabling it would also break logging. Tampering is visible.
Threat model recap: Layer 2 alone is sufficient to block cross-tenant data access. Layers 1, 3, and 4 add early feedback, defense in depth, and catch misconfiguration. If the Postgres role grants are ever wrong, L3's locked search_path still blocks naive unqualified queries; L4 catches the Python paths that try to rewrite them. We treat L2 as the load-bearing layer and audit it in CI: a startup check verifies grants for every tenant role at boot.

09 · Observability

Logs, prints, and history

Three independent stores feed the UI: structured rows on CodeExecution (status, duration, return value, traceback), line-by-line rows on CodeExecutionLogLine (live tail), and the existing auditlog (who did what when).

▤

Live tail

UI polls log-tail/?after_seq=N while status is non-terminal. Returns only new lines. Switches to a static fetch once the run lands in a terminal state.

∎

Captured streams · print & logger

Both are captured. print() → stdout level. logging.info / warning / error / debug → a dedicated codexec logging handler installed during the run that routes by LogRecord.levelname (INFO / WARN / ERR) and writes each record as a structured line. Tracebacks from raised exceptions are appended at ERR level. Django ORM query logs are suppressed by default; toggle via codexec.debug_sql = True.

▣

OpenTelemetry

Spans emitted for the celery task and inner exec. Tags: tenant, snippet_id, version, duration_ms, status. Hooks into existing telemetry pipeline.

10 · Files

File I/O — inputs & outputs

Snippets often need to read a CSV/JSON/XLSX and produce a result file. The runtime exposes a codexec helper module that surfaces a per-run sandbox: uploaded files are staged at codexec.inputs / <name>; anything the script writes under codexec.outputs is auto-collected, stored in the object store, and listed for download from the run history.

The codexec file API

A small set of helpers injected as the zango.codexec module. The user never touches S3, never builds a path, never opens a file at an absolute location on the worker disk. Every read and write goes through the file-mapping model, so every byte the script consumes or produces is tracked.

The file mapping model

Every file the script touches — input or output — is a row in CodeExecFile. The name the script uses (e.g. "report.csv") is the row's logical key. The actual bytes live wherever Django's STORAGES['default'] points (S3 / GCS / local MEDIA_ROOT). The script doesn't know or care.

! Filename preservation — design rule

Storage backends (especially S3 + Django's FileField) frequently rewrite filenames to avoid collisions or sanitize characters. We must not depend on the filename round-tripping through storage. The rule:

  • Source of truth = CodeExecFile.name — the original (sanitized) filename the user uploaded or the script wrote. Stored explicitly in the DB. Never derived from .file.name at runtime.
  • Storage key uses a UUID, not the name — <tenant>/codexec/<exec_id>/<kind>/<file_uuid>. No extension on disk. Collisions impossible regardless of what users upload (two report.csv uploads in the same run still get unique storage keys; DB unique constraint on (execution, kind, name) rejects the second).
  • Lookups by name go to the DB — codexec.read("foo.csv") does CodeExecFile.objects.get(execution=…, kind="input", name="foo.csv") then opens obj.file. The storage backend never sees the name.
  • Downloads restore the original name — the download endpoint sets Content-Disposition: attachment; filename="<CodeExecFile.name>" (RFC 5987 encoded for unicode). For S3-signed URLs, we set response-content-disposition on the presign. User downloads report.csv regardless of what S3 has on disk.
  • Name sanitization on entry — strip null bytes, normalize NFC, reject .. and absolute paths, take only the basename, cap at 255 chars. Done once at upload / first write; stored value is final.

Implementation: FileField's upload_to returns the UUID path; we ignore file.name as a source of identity entirely.

How it works end to end

⌂

Save · snippet-level files

Files uploaded in the editor stream to storage at <tenant>/codexec/snippets/<snippet_id>/<file_uuid>, then one CodeSnippetFile row per file. These persist with the snippet and re-appear next time the modal is opened. No run is created.

⇉

Run · snapshot & execute

For each CodeSnippetFile, insert CodeExecFile(execution=new, kind=input, source_snippet_file=<ref>) pointing at the same blob (no byte copy). Then enqueue the celery task. After this point the run carries its own complete file set — editing the snippet's files later cannot rewrite this run's history.

↥

During run · read & write

codexec.read("foo.csv") → DB lookup CodeExecFile(execution=this, kind=input, name="foo.csv") → stream from storage.
codexec.write("bar.csv", data) → stream to storage → insert CodeExecFile(kind=output). Outputs are always execution-level.

Limits & lifecycle: Per snippet: ≤ 10 input files, ≤ 25 MB each, ≤ 100 MB total. Per run output: ≤ 50 files, ≤ 100 MB total. Unique (snippet, name) on snippet files; unique (execution, kind, name) on exec files. Snippet files live as long as the snippet; exec files retained 90 days then GC'd. Storage blobs are reference-counted — a snippet file's blob is only deleted once no execution still references it.

✕ Removing a file from a snippet — what happens

When the user clicks the × on a file in the editor modal and saves:

  • The CodeSnippetFile row is hard-deleted. No is_deleted flag, no tombstone. The editor will not show it anymore, and future runs of this snippet will not snapshot it.
  • Past CodeExecFile rows are untouched. They have their own row tied to their own execution. The source_snippet_file FK is declared on_delete=SET_NULL, so it cleanly becomes null — past runs still list the file, still let you download it, just without a provenance pointer back to the (now-gone) snippet file.
  • The storage blob stays. Snippet file and exec file rows share a storage_key. The GC job deletes the blob only when no row (snippet-level or exec-level) references it anymore — typically when the last referring execution ages out of the 90-day window.
  • Add a new file with the same name? Just works. Unique constraint is (snippet, name) on live rows only (hard-deleted rows aren't there). The new upload gets a fresh UUID storage key and its own blob.
Path safety: name is normalized — no .., no leading slash, no absolute paths. Only the basename is kept. So codexec.write("../../etc/passwd", ...) either errors or just creates a file literally named etc_passwd under this run's output prefix. Cross-run / cross-tenant access by guessing names is impossible: the lookup is keyed on the current execution_id (only the executor has it).

10 · Robustness

Edge cases & how we handle them

The unhappy paths — what the executor must survive without leaking state.

⏱
Run exceeds timeout
Default 60 s · max 300 s · configurable per snippet via timeout_seconds. Enforced two ways: Postgres statement_timeout for single queries, and celery soft_time_limit for total wall-clock. On hit, celery raises SoftTimeLimitExceeded; captured streams flushed, status → timeout, partial stdout preserved. Hard kill at +5 s.
↺
Concurrent run on same snippet
API checks for an in-flight execution; returns 409 with the in-flight execution id. UI suggests "Watch running execution" instead.
⚠
Worker crashes mid-run
Beat sweeper job marks executions stuck in running with no heartbeat for >2× timeout as failed · worker_lost.
∞
Infinite stdout
After 1 MB total / 8 KB per line, writes become no-ops; a final SYS line records the truncation. Run continues to allow it to terminate via timeout.
↷
Snippet edited mid-run
Edits bump the snippet version. The in-flight run executes its snapshot, unaffected. New run uses the new version.
⊘
Deny list updated after save
Re-validation at job pickup catches retroactively-banned tokens. Execution fails before the user code runs, status=failed · denylist.
✕
Snippet deleted with history
Delete is soft (is_archived=True). History remains queryable, source_snapshot lives on the execution rows.
⤓
Large return value
Return value is JSON-serialized and capped at 64 KB. Oversize → stored as truncated + return_value_truncated=True; full value available via stdout if user printed it.
⛨
Attempted cross-tenant query
Caught at Layer 1 (AST) or Layer 2 (Postgres role). On L2 hit, Postgres returns permission denied for schema tntApp — the executor catches it, marks status failed · isolation_violation, and writes a SECURITY-level audit log row tagged with snippet id, source hash, and the offending query.
