Zango platform · exports + my downloads

Users and logs, exported — asynchronously and safely capped.

Operators can now export any list surface (App Users, Access Logs, Audit Logs) as CSV via a small Export button next to the search box. Jobs run through Celery under the existing tenant-suspend guard. A new "My Downloads" page lists every job with its filters, status, and a per-row Download / Delete. Files live for as long as the operator wants them — no auto-expiry, per-row control instead.

4 export kinds, 1 model, 1 downloads page 30,000-row cap · pre-dispatch validation CSV only · streaming · no third-party deps

Why async when 30k rows fits in 8 seconds

Even though every export could finish sync-fast under the cap, async gives us: (a) a persistent audit trail of who exported what with what filters — real value on a multi-tenant pharma platform, (b) tenant-suspend guard for free via the Celery chokepoint, (c) a UX where the operator clicks once and comes back later, and (d) parity with codexec's job pattern that operators already understand.

Design decisions

One-line filter summary beats a JSON blob — operators scan by "was this the one I ran with the timestamp filter?" not by reading dictionaries. Manual refresh button because we agreed no polling; a small pulse animation on the Running dot signals "still working". Delete on Failed rows stays enabled so the user can clean up their history. No expiry column since we don't auto-expire; the column set stays tight.

05 · Backend model

One table for all four kinds

Lives in the new zango.apps.exports app. Tenant-scoped (in TENANT_APPS). All four export types share this schema — kind is a discriminator field.

backend/src/zango/apps/exports/models.py new 
from uuid import uuid4
from django.db import models

from zango.core.model_mixins import FullAuditMixin
from zango.core.storage_utils import ZFileField


class ExportJob(FullAuditMixin):
    """A user-requested CSV export of one of the platform list surfaces."""

    class Kind(models.TextChoices):
        APP_USERS            = "app_users",            "App Users"
        ACCESS_LOGS          = "access_logs",          "Access Logs"
        AUDIT_LOGS_APP       = "audit_logs_app",       "Application Audit Logs"
        AUDIT_LOGS_FRAMEWORK = "audit_logs_framework", "Framework Audit Logs"

    class Status(models.TextChoices):
        QUEUED  = "queued",  "Queued"
        RUNNING = "running", "Running"
        SUCCESS = "success", "Success"
        FAILED  = "failed",  "Failed"

    object_uuid   = models.UUIDField(default=uuid4, unique=True, editable=False)
    kind          = models.CharField(max_length=32, choices=Kind.choices)
    status        = models.CharField(max_length=16, choices=Status.choices, default=Status.QUEUED)

    # Full filter payload captured at request time. Used by the celery
    # builder to reconstruct the queryset AND rendered on My Downloads.
    filters       = models.JSONField(default=dict)
    # Server-built one-line summary of `filters` for the UI.
    filters_summary = models.CharField(max_length=512, blank=True, default="")

    row_count     = models.PositiveIntegerField(null=True, blank=True)
    file          = ZFileField(null=True, blank=True)
    filename      = models.CharField(max_length=256, blank=True, default="")
    size_bytes    = models.PositiveBigIntegerField(null=True, blank=True)

    error_message   = models.TextField(blank=True, default="")
    celery_task_id  = models.CharField(max_length=64, blank=True, default="")

    requested_by  = models.ForeignKey(
        "platformauth.PlatformUserModel",
        on_delete=models.SET_NULL,
        null=True,
        related_name="+",
    )
    started_at   = models.DateTimeField(null=True, blank=True)
    completed_at = models.DateTimeField(null=True, blank=True)

    class Meta:
        ordering = ["-created_at"]
        indexes = [
            models.Index(fields=["requested_by", "-created_at"]),
            models.Index(fields=["kind", "status"]),
        ]

Terminal statuses are SUCCESS and FAILED. The rate-limit check queries status__in=[QUEUED, RUNNING].

06 · API endpoints

Four routes, one router

RoutePurpose
POST /api/v1/apps/<uuid>/exports/<kind>/ Create a new export job for the given kind. Body carries the filter payload. Returns 201 with job details, 400 on cap exceed, 409 on rate-limit hit.
GET /api/v1/apps/<uuid>/exports/ List all jobs the current user has requested for this app. Paginated. Newest first.
GET /api/v1/apps/<uuid>/exports/<job_uuid>/ Detail — used for the failed-job "view error" popover.
GET /api/v1/apps/<uuid>/exports/<job_uuid>/download/ Stream the CSV. Only when status=success and file is populated. Returns a signed URL when S3, direct stream when local FS.
DELETE /api/v1/apps/<uuid>/exports/<job_uuid>/ Delete a job (row + file). Cancel semantics when the job is still running: sets a "cancelled" flag the celery task polls.

Rate-limit + cap enforcement

backend/src/zango/apps/exports/views.py — ExportJobCreateView new 
def post(self, request, app_uuid, kind):
    # 1 — Validate kind
    if kind not in ExportJob.Kind.values:
        return get_api_response(False, {"message": f"Unknown export kind '{kind}'"}, 400)

    # 2 — Rate limit: one QUEUED/RUNNING per (user, kind)
    active_qs = ExportJob.objects.filter(
        requested_by=request.user.platform_user,
        kind=kind,
        status__in=[ExportJob.Status.QUEUED, ExportJob.Status.RUNNING],
    )
    if active_qs.exists():
        return get_api_response(
            False,
            {"message": "An export of this kind is already in progress. "
                        "Check My Downloads."},
            409,
        )

    # 3 — Pre-count against the actual queryset the builder will use
    filters = request.data.get("filters", {})
    builder = EXPORT_BUILDERS[kind]
    row_count = builder.count(filters)   # fast COUNT(*)

    if row_count > MAX_EXPORT_ROWS:  # MAX_EXPORT_ROWS = 30_000
        return get_api_response(
            False,
            {
                "message": f"Too many rows to export. Your filters match "
                           f"{row_count:,} rows (max {MAX_EXPORT_ROWS:,}). "
                           f"Try adding a date range or narrower search.",
                "row_count": row_count,
                "max_rows": MAX_EXPORT_ROWS,
            },
            400,
        )

    # 4 — Create + dispatch
    job = ExportJob.objects.create(
        kind=kind,
        filters=filters,
        filters_summary=builder.summarise(filters),
        row_count=row_count,
        requested_by=request.user.platform_user,
    )
    # Route through zango_task_executor so suspend guard applies.
    run_export.apply_async(args=[tenant.name, str(job.object_uuid)])

    return get_api_response(True, {"job": ExportJobSerializer(job).data}, 201)
07 · The Celery task

Streams rows through a builder and writes to storage

backend/src/zango/apps/exports/tasks.py new 
@shared_task(bind=True, name="zango.exports.run_export")
def run_export(self, tenant_name, job_uuid):
    from zango.apps.shared.tenancy.utils import assert_tenant_active

    tenant = TenantModel.objects.get(name=tenant_name)
    connection.set_tenant(tenant)
    assert_tenant_active(tenant)   # existing framework guard

    job = ExportJob.objects.get(object_uuid=job_uuid)
    job.status = ExportJob.Status.RUNNING
    job.started_at = timezone.now()
    job.celery_task_id = (self.request.id or "")[:64]
    job.save(update_fields=["status", "started_at", "celery_task_id", "modified_at"])

    try:
        builder = EXPORT_BUILDERS[job.kind]
        with tempfile.NamedTemporaryFile(mode="w", suffix=".csv", delete=False) as tmp:
            row_count = builder.write_csv(job.filters, tmp)
        with open(tmp.name, "rb") as fp:
            job.file.save(f"exports/{job.object_uuid}.csv", File(fp))
        os.unlink(tmp.name)

        job.filename   = f"{job.kind}-{timezone.now():%Y%m%d-%H%M%S}.csv"
        job.row_count  = row_count
        job.size_bytes = job.file.size
        job.status     = ExportJob.Status.SUCCESS
    except Exception as exc:
        log.exception("export job %s failed", job_uuid)
        job.status = ExportJob.Status.FAILED
        job.error_message = str(exc)[:1000]
    finally:
        job.completed_at = timezone.now()
        job.save()
Why .iterator(chunk_size=1000) matters

Each builder uses qs.iterator(chunk_size=1000) so Django never materialises the whole queryset in memory. Even 30k audit-log rows with big changes JSON columns stay under 50 MB peak worker RAM. Without this, one large export would OOM the worker.

08 · Builders

Four small classes, one interface

Each builder wraps the existing get_queryset logic from the list view so the export filters match the browse UI byte for byte — no drift possible. Same search, same column filters, same behaviour.

backend/src/zango/apps/exports/builders.py new 
class ExportBuilder:
    """Common interface for all export kinds."""

    def queryset(self, filters):
        raise NotImplementedError

    def count(self, filters) -> int:
        return self.queryset(filters).count()

    def summarise(self, filters) -> str:
        """One-line summary rendered on My Downloads."""
        raise NotImplementedError

    def write_csv(self, filters, fp) -> int:
        writer = csv.writer(fp)
        writer.writerow(self.HEADERS)
        count = 0
        for row in self.queryset(filters).iterator(chunk_size=1000):
            writer.writerow(self.row_for(row))
            count += 1
        return count


class AppUsersBuilder(ExportBuilder):
    HEADERS = ["ID", "Name", "Email", "Mobile", "Roles", "Active", "Last Login", "Created"]

    def queryset(self, filters):
        from zango.api.platform.tenancy.v1.views import UserViewAPIV1
        view = UserViewAPIV1()
        return view.get_queryset(
            filters.get("search", ""),
            filters.get("columns", {}),
        )

    def summarise(self, filters):
        parts = []
        if filters.get("search"):
            parts.append(f'search "{filters[\"search\"]}"')
        if filters.get("columns", {}).get("is_active"):
            parts.append("active only" if filters["columns"]["is_active"] == "true" else "inactive only")
        return " · ".join(parts) or "no filter"

    def row_for(self, u):
        return [
            u.id, u.name, u.email, u.mobile,
            ", ".join(r.name for r in u.roles.all()),
            "Yes" if u.is_active else "No",
            u.last_login.isoformat() if u.last_login else "",
            u.created_at.isoformat(),
        ]


# AccessLogsBuilder, AuditLogsAppBuilder, AuditLogsFrameworkBuilder
# — all follow the same shape.

EXPORT_BUILDERS = {
    ExportJob.Kind.APP_USERS            : AppUsersBuilder(),
    ExportJob.Kind.ACCESS_LOGS          : AccessLogsBuilder(),
    ExportJob.Kind.AUDIT_LOGS_APP       : AuditLogsAppBuilder(),
    ExportJob.Kind.AUDIT_LOGS_FRAMEWORK : AuditLogsFrameworkBuilder(),
}

Zero drift: because each builder calls the list view's get_queryset directly, whatever the operator sees in the browse UI is exactly what they get in the CSV. When we fix a filter bug in one place, both surfaces get the fix automatically.

11 · Edge cases

What happens when…

The app is suspended while an export is running?

The Celery task routes through zango_task_executor, which calls assert_tenant_active() at pickup. If the flip happened mid-queue, the task raises TenantSuspended before doing any work; the finally-block marks the job failed with a clear message. Same pattern codexec already uses.

The user closes the tab mid-export?

Nothing to worry about. The Celery task keeps running; the row moves through RUNNING → SUCCESS in the background. The file is waiting on My Downloads next time they visit.

The user deletes a job that's still Running?

DELETE marks the row for cancellation. The celery task polls its own row every ~1000 iterations and bails if the row is missing or marked cancelled. Storage cleanup happens automatically.

A superadmin views another user's downloads?

Scoped by requested_by. My Downloads only shows your own jobs. If a cross-user view is needed later, we add a superadmin-only "All exports" tab — not in v1.

The pre-count says 29,998 but the actual query returns 30,003 by the time the worker runs?

Rare — happens only if new rows landed between count and execution. The builder still writes them all; the CSV is a snapshot of what matched when the worker ran. We don't re-check the cap in the worker to avoid failing jobs the user already committed to.

The CSV contains PII — is that logged anywhere?

The ExportJob row is auditable (via existing FullAuditMixin) — creation, edits, deletion tracked. The file contents themselves aren't logged. Consider adding an auditlog.register(ExportJob) line as a follow-up so exports show up in the audit log.
