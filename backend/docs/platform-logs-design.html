Overview

What we're building

A Logs tab in the platform app panel that fetches and displays logs from wherever they happen to live for the current environment. The plumbing that ships them there (CloudWatch Agent, awslogs driver, OTel exporter, Lambda forwarder, etc.) is owned by infra — Zango treats those stores as read-only sources via a connector interface.

The status quo

  • Containers ship stdout/stderr to CloudWatch via the ECS awslogs driver.
  • One log group per (env × component): /ecs/<env>/{app,celery,celery-beat,nginx}.
  • Local dev: logs land in the terminal where you ran runserver / celery worker.
  • To investigate anything, devs SSH into ECS or open CloudWatch in the AWS console.

What good looks like

  • Logs tab in the app panel reads live from CloudWatch (or whatever store each component is wired to).
  • Configurable per component — app can be on CloudWatch, celery on DynamoDB, beat on OTel.
  • Search, filter, live tail, time-range, raw FilterLogEvents pattern for power users.
  • Future: tenant-scoped filtering once the producer side emits structured logs.
What changed

From write-side capture to read-side connector

Earlier sketches of this feature had Zango installing a Python logging.Handler that wrote into a Postgres table. That coupled the app server to log storage and forced every customer to operate Postgres for logs. The shift below decouples Zango from the storage choice entirely.

Dimension Old (write-side capture) New (read-side connector)
Who ships logs Zango — custom handler writes to Postgres. Infra — existing pipelines (awslogs driver, OTel collector, etc.). Out of scope here.
Storage choice Hard-coded Postgres. Pluggable — CloudWatch (v1), DynamoDB, OTel backends, others.
Per-component flexibility One model, all sources. Each (env × component) row picks its own connector + config.
Write amplification risk Hot-tenant write storm in Postgres. None — Zango doesn't write logs.
Tenant filtering Native (schema isolation). Depends on the producer adding a tenant_schema field — see §3.

The tenant gap

Tenant isolation in the current log format

Updated assessment · the infrastructure exists; it's only wired to the wrong handler.

What already exists in the codebase — found in zango/config/settings/base.py and zango/core/monitoring/log_filter.py:

  • TenantContextFilter reads connection.tenant.schema_name and domain_url and decorates every record with app_name + domain_url.
  • A verbose formatter already embeds those fields: [%(app_name)s:%(domain_url)s][%(asctime)s] %(levelname)s ….
  • The filter + verbose formatter are attached only to a rotating file handler, and only when a log-folder env is configured.
  • The default console handler — the one ECS ships to CloudWatch — uses the plain console formatter without the filter, hence the unstructured lines we see in CloudWatch.

Implication. The "producer-side fix" is not new plumbing — it's ~5 lines of settings to attach the existing filter + verbose formatter to the console handler. That makes shipping tenant-scoped logs from v1 plausible (see updated Phasing).

What can be inferred today (best-effort, not recommended for the UI)

  • App: WSGIRequest repr in tracebacks contains URL path — sometimes carries an app UUID under /platform/apps/<uuid>. Not stable; not present on INFO.
  • Celery: task name is logged, tenant is not. Some task argument logs accidentally leak tenant context, but not consistently.
  • Nginx (separate log group): would have the Host header which maps to a tenant — but Nginx is out of scope for v1.

How the design handles the gap

Option What the user sees What it costs
A · ship as-is All lines for the selected (env × component), no tenant filter, with a banner: "Tenant-scoped log filtering not yet available — producer-side change required." Logs tab gated to platform admins. Zero producer change. v1 ships.
B · land producer fix with v1 (recommended) Tenant admin sees only their tenant's lines from day one. ~5 LOC of settings — attach the existing tenant_filter to the console handler, swap formatter to verbose in staging/prod, wire Celery's setup_logging. The connector adds [<schema>: to every FilterLogEvents pattern (verbose) or { $.app_name = … } (if we go JSON).

Given the producer side is essentially free (the filter is already written), Option B is the recommended path. v1 ships tenant-scoped on day one; the banner-and-block-tenant-admins contingency in Option A only comes into play if the producer change blocks for some reason.

v1 ships with the gap explicit. The Logs tab is gated behind a platform.logs.view policy that maps only to platform admins until v2 lands — see § Permissions.

Why we don't try to parse tenants out of plain-text lines. Heuristic parsing (URL regex, WSGIRequest repr matching) is fragile and will both miss lines and leak lines. A banner telling the user "we don't have this yet" is more honest than a filter that silently produces wrong results.
Abstraction

The LogConnector interface

A small Python protocol. Each concrete connector implements the same four operations; the views and the frontend don't care which one is wired up.

pyzango/apps/platform_logs/connectors/base.py
class LogConnector(Protocol): """Read-only interface for fetching log lines from an external store.""" def list_streams(self, *, since: datetime) -> List[LogStream]: ... # Discover the sub-streams (CW log streams, OTel resource streams, etc.) # that exist under this connector since `since`. Powers the optional # stream-picker in the UI. def fetch(self, *, filters: LogFilters, page: Cursor) -> LogPage: ... # Return a bounded page of lines matching `filters`. Backward # pagination via Cursor.before, forward (tail) via Cursor.after. def facets(self) -> FacetSet: ... # Return values for UI filter dropdowns: known log levels (from # FilterLogEvents stats), recently active streams. Cached 60s. def deep_link(self, *, filters: LogFilters) -> Optional[str]: ... # Build a "Open in native UI" URL — e.g. a CloudWatch Logs Insights # console link pre-filtered by the same filters. Optional.

Common types

Config storage

Connector configuration

Configuration lives in the public schema as a single model. One row per (env × component). A platform admin edits these from a platform-admin page; no per-tenant override in v1 (all tenants in an env share the same wiring).

pyapps/platform_logs/models.py

Lives in public schema.""" id = models.UUIDField(primary_key=True, default=uuid.uuid4) environment = models.CharField(max_length=32) # "staging", "prod", "sandbox", "local" component = models.CharField(max_length=16, choices=Component.choices) connector = models.CharField(max_length=32, choices=ConnectorType.choices) config = models.JSONField() # type-specific blob; see CW config below is_active = models.BooleanField(default=True) class Meta: db_table = "platform_log_connector_config" constraints = [ models.UniqueConstraint( fields=["environment", "component"], name="unique_env_component", ), ]

Resolution at request time

  • The Logs view picks the active environment from settings.ENV (already in the codebase).
  • For each component the user clicks on, look up the matching row, instantiate the corresponding connector class with the row's config dict.
  • If no row exists for (env, component), the UI shows an empty state with a "Configure connector" link (no crash, no error).

v1 connector

CloudWatch connector

Single concrete implementation in v1. Uses boto3 against the logs service. No cross-account assumption in v1; uses the IAM role attached to the Zango ECS task / EC2 / dev environment.

Config shape

FieldTypeMeaning
regionstringAWS region of the log group. Required. E.g. ap-south-1.
log_group_namestringFull CloudWatch log group name. Required. E.g. /ecs/zango-india-staging/app.
stream_prefixstring · optionalIf set, restrict reads to streams whose name starts with this prefix. Useful when one group is shared across multiple services.

API calls used

Connector opAWS callNotes
list_streams(since) logs:DescribeLogStreams Order by LastEventTime DESCENDING; filter client-side on lastEventTimestamp >= since. Paginate with nextToken. Cached 30s.
fetch(filters, page) — default logs:FilterLogEvents The workhorse. Fans out across all streams in the group automatically; supports a CloudWatch filter pattern + time window + nextToken. No per-stream coordination needed in our code.
fetch(filters, page) — stream-restricted logs:FilterLogEvents with logStreamNames=[…] When the UI's stream-picker is used, pass only the selected streams. Up to 100 streams per call.

facets() logs:DescribeLogStreams + logs:FilterLogEvents sample Sample 200 recent lines, parse out distinct levels client-side. Cached 60s.

Level & timestamp parsing (v1 · plain-text)

Since lines today are unstructured, the connector applies a tiny regex pass to lift a level when possible and uses the CloudWatch event timestamp as ts. Unparseable lines still display with the right timestamp; level shows as —.

py
# Celery default: "[2026-06-01 08:06:00,405: INFO/MainProcess] …" CELERY_RE = re.compile(r"\[\d{4}-\d{2}-\d{2}[^\]]+:\s*(?P<lvl>DEBUG|INFO|WARNING|ERROR|CRITICAL)/") # Python stdlib default: "INFO 2026-06-01 08:06:00 module: msg" STDLIB_RE = re.compile(r"^(?P<lvl>DEBUG|INFO|WARNING|ERROR|CRITICAL)\s")
Streams

Handling one-or-many streams per log group

A log group routinely has 5–50 streams (one per running ECS task plus deploy history). The connector handles this transparently for the default browse path and exposes streams explicitly only when the user wants to drill in.

Default browse — group-wide

  • FilterLogEvents with just logGroupName automatically fans out across all streams in the group and merges events by timestamp.
  • No per-stream code in our connector. Single API call per page.
  • Result lines carry their originating logStreamName; the UI shows a small "stream" badge so users can tell which task instance produced a line.

Optional stream picker

  • UI offers a "Streams" dropdown populated from DescribeLogStreams (default: streams with events in the last 24h).
  • Selecting one or more streams passes logStreamNames=[…] to FilterLogEvents.
  • Useful when investigating a specific task instance ("which container served this request?").

Live tail behavior across streams

  • Tail polls FilterLogEvents with startTime = lastSeenTs + 1ms every 2 seconds.
  • Group-wide tail naturally picks up events from newly-spawned streams (new ECS tasks during a deploy) without any reconfiguration.
  • Per-stream tail is bounded by the picker: if the user pinned a specific stream and it goes away (task replaced), the UI shows a "stream ended" indicator and offers to switch back to group-wide.

Pagination & cursors

DirectionHow
Backward · "Load earlier" Issue FilterLogEvents with endTime = oldestSeenTs - 1ms. Continue until empty.
Forward · live tail startTime = newestSeenTs + 1ms on each poll.
Within a page CloudWatch's nextToken chained until exhausted or limit hit.
CloudWatch quirk · eventual ordering across streams. FilterLogEvents merges streams by event timestamp, but CloudWatch can deliver lines slightly out-of-order across streams (ingestion lag varies per task). The UI surfaces a small "ingestion lag" hint when the newest-line ts trails wall-clock by >5s, so users don't think the tail is broken.

Common query params (browse / tail)

qs
?since=2026-05-29T11:00:00Z # required; defaults to 1h ago in the UI &until=2026-05-29T12:00:00Z # browse only; tail uses "now" &q=traceback # free-text search &pattern=%5BERROR%5D # native CW filter pattern, URL-encoded &levels=ERROR,WARNING # client-side post-filter (plain-text logs) &streams=app/app/abc123,app/app/def456 &limit=200 &before=<cursor> # backward pagination &after=<cursor> # forward (tail) pagination

Throttling

  • Per-user rate limit: 30 requests / minute on browse, 60 / minute on tail.
  • Connector calls are tagged with a request-id passed to boto3 so AWS-side throttling can be correlated.
  • If CloudWatch returns ThrottlingException, the API returns 503 with Retry-After; the frontend backs off the tail.
Frontend

App-panel UI

A platform-level Logs tab, sibling to Code Execution. Component picker at the top, virtualized log feed below, filter rail on the right.

Header strip

  • Component segmented control: App · Celery · Celery beat. Reflects which (env × component) rows exist; greys out unconfigured ones.
  • Time-window picker: 15m · 1h · 6h · 24h · custom.
  • Search input: free-text substring (passed as q; server-side for CW it becomes part of the filter pattern).
  • Live tail toggle.
  • "Open in CloudWatch" link: uses the /deep-link/ endpoint to open AWS console with the same filters.

Filter rail (right)

  • Levels — multi-toggle. Note: until producer-side fix lands, level is parsed best-effort from the line text; some lines won't have a level.
  • Streams — searchable list. Default empty (group-wide). Selecting one or more restricts FilterLogEvents to those streams.
  • Native pattern — power-user field; raw CloudWatch filter syntax accepted (e.g. [ERROR], "DoesNotExist").

Access control

Permissions & policies

Two new policies, mapped per phase to match the tenant-filtering reality.

Policy Grants v1 mapping v2 mapping
platform.logs.view View the Logs tab inside an app; browse + tail that app's lines. Platform admin only (UI cannot yet filter to a single app). App users with access to that app — read filtered by app_name at connector time.

Local dev

Parity story

Local dev has no CloudWatch. v1 ships without a local connector — the Logs tab shows a friendly empty state on ENV=local with a "Logs are viewed in your terminal locally" message. A future FileTailConnector reading rotating log files is an obvious follow-up but out of v1 scope.

What this means in practice

  • No new infra for local dev to maintain.
  • Connector + admin UI still work locally so the feature can be tested end-to-end against a staging connector config row.
  • If a dev does want local logs in the panel, they can point a local connector row at a staging CloudWatch group (read-only, same IAM role).
Producer-side fix (v2)

Closing the tenant gap

Separate PR after v1 ships. Owned by backend; this section is here so the spec hangs together.

What's already in place

From zango/core/monitoring/log_filter.py — already implemented, already registered as the tenant_filter in the LOGGING dict:

pyexisting · log_filter.py
class TenantContextFilter(logging.Filter): def filter(self, record): record.app_name = connection.tenant.schema_name record.domain_url = getattr(connection.tenant, "domain_url", "") # ... exc_traceback_content, pathname tidy-up ... return True

The change (small)

  1. Attach tenant_filter to the console handler in base.py LOGGING dict — currently only the rotating-file handler has it.
  2. Switch console formatter from console to verbose in staging + prod (so the [app_name:domain_url] prefix lands on every line ECS ships to CloudWatch). Keep console formatter for local dev.
  3. Add a JSON formatter alongside verbose. Recommended for v2-of-v2: structured JSON makes the CloudWatch $.app_name = "…" filter pattern trivially precise. Optional for tenant filtering — the verbose bracketed format is parseable with a small regex, JSON is cleaner.
  4. Wire the Celery worker's logger config (currently inherits Django root) to also attach the tenant_filter + verbose / JSON formatter, ideally via celery's setup_logging signal so it doesn't auto-revert.
pysettings/base.py — proposed diff (~5 LOC)
# in LOGGING["handlers"]["console"]: { "class": "logging.StreamHandler", "formatter": "verbose" if ENV in {"staging", "prod"} else "console", "filters": ["tenant_filter"], # <-- new line }

Celery — note

Celery's default behaviour is to replace Django's logging config with its own at worker boot. Connect Zango's LOGGING dict via the @setup_logging.connect signal so the tenant filter survives worker startup. Without this, celery logs in CloudWatch will keep the old plain-text format even after the producer fix.

After it ships

  • Every line in CloudWatch starts with [<schema>:<domain>] (verbose) or carries an app_name field (JSON).
  • Connector config row: set format to verbose (regex-parsed) or json.
  • FilterLogEvents pattern: [<schema>: for verbose; { $.app_name = "<schema>" } for JSON.
  • UI banner removed; platform.logs.view opens up to tenant admins.
Already-installed OTel context. The codebase also has OtelZangoContextMiddleware and OTLP exporter settings (gated by OTEL_IS_ENABLED). If staging or prod is already exporting traces, log records likely already carry trace_id / span_id. A future OTel connector could use those for cross-system correlation.
Risks

Risks & guardrails

Risk Mitigation
CloudWatch API quota / cost
spend		 FilterLogEvents is billed per scanned GB. Cache facets 60s, cap browse window to 24h by default, throttle per-user request rate, surface "request count" in dev tools for visibility.
IAM scope creep
security		 Per-environment ARN list in the IAM policy. No logs:* wildcard. Cross-account reads via role_arn deferred to v2 with explicit STS audit.
Plain-text level parsing miss
UX		 Regex falls back gracefully — unparsed level shows as —. Documented; users learn to rely on free-text search until v2.
Stream churn during deploy
UX		 Group-wide default ignores the new stream — events still flow in. Stream-pinned views show "stream ended" when the task is replaced and offer to switch back.
Tenant info leak (v1)
privacy		 v1 is platform-admin-only by policy. Tenant admins explicitly cannot reach the Logs tab until v2 lands.
Connector config drift
ops		 Admin UI validates: log group exists (DescribeLogGroups), IAM call succeeds with a sample DescribeLogStreams. Saves are atomic; failed validation blocks save.
Boto3 import / start-up cost
perf		 boto3 client created lazily inside the connector class and memoized per (region, role). No global boto3 import outside the platform_logs app.
Throttling / TooManyRequestsException
UX		 Connector wraps every call in a retry-with-jitter (3 attempts) for ThrottlingException / ServiceUnavailable. Beyond that, propagate to UI with backoff.
Future connector explosion
debt		 Strict Protocol; connectors live under one folder; each new type requires a test class that hits a fake / replay fixture before merge.

Open

Questions for the team

  • Confirm CloudWatch is the right v1 connector for both staging and prod (and that no env currently ships logs elsewhere).
  • Should the connector config live in the public schema or per-tenant? Spec assumes public; per-tenant would let different tenants point to different stores but adds setup burden.
  • Cross-account reads: is BMS prod (/ecs/zango-bms-india-prod/…) the same AWS account as the Zango ECS service? If not, we need role_arn + STS in v1 instead of v2.
  • Retention / archival on the CloudWatch side — keep current per-group retention, or align them now?
  • Confirm Option B (land producer fix with v1) — it's small and unlocks tenant scoping from day one. Any reason to defer to a follow-up PR?
  • Verbose bracketed format vs JSON for staging/prod console — verbose is parseable today, JSON is cleaner long-term. Pick one for the producer fix.
  • Is OTel already exporting logs (not just traces) in staging/prod? If yes, an OTel/OTLP read connector becomes a viable alternative to CloudWatch sooner than v6+.
  • Should the admin UI live under "Platform settings" or be a sub-tab of the Logs tab itself?
  • Do we want NDJSON export from the connector (limited to N lines) for sharing snapshots in tickets, or is the deep-link enough?
