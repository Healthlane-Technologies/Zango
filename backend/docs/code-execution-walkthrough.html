01 · Overview

What this feature does

A platform-managed catalog of Python snippets. Each snippet can be edited in a Monaco editor, has input files attached to it, and can be triggered to run inside the tenant's Celery worker — with stdout/stderr captured, files produced auto-registered, and every run kept as a self-contained history row.

▸ Save & Run

UI saves the snippet (creates the row + uploads any files), then triggers a Celery task that runs the source in the tenant's workspace.

◐ Tracked history

Every run is a frozen row: source snapshot, captured stdout/stderr, per-line log table, return value, files in & out.

⛨ Defense in depth

AST deny list (L1) at save + pickup; locked search_path + statement_timeout (L3); ConnectionGuard against reflective tenant-switching (L4).

1 Multi-tenancy & schemas

Zango hosts many apps ("tenants") on one server. Each tenant gets its own Postgres schema — a namespace of tables. So a patient table can exist in both bmspspmonitoring and tntApp schemas without conflict.

Django talks to Postgres through one connection at a time. When a request comes in, django-tenants figures out which tenant it's for and runs SET search_path TO "bmspspmonitoring", public so all queries hit that tenant's tables.

2 Celery & workers

Celery is a background-job system. Your web request says "do this slow thing later" by calling my_task.apply_async(args=[…]). The args get serialized and pushed to a message broker (Redis or RabbitMQ).

A separate process — the "worker" — pulls jobs off the broker and runs them. When you change tasks.py, the worker is using the old version until you restart it. That's why the operating notes say "restart celery after backend changes".

3 Django models & the ORM

A model class subclasses models.Model. Each class attribute is a column. Meta inside the class controls table name, ordering, indexes, constraints. Manager (the .objects attribute) is how you query: CodeSnippet.objects.filter(is_archived=False) turns into a SQL SELECT.

A ForeignKey is a column that references another table. on_delete=CASCADE means "if the parent row is deleted, delete this one too"; on_delete=PROTECT blocks parent deletion while children exist; SET_NULL just nulls the FK column.

4 Migrations

Files inside app/migrations/ that describe schema changes. Running manage.py makemigrations compares your models.py to the previous state and writes a new migration. manage.py migrate applies them to the database.

In Zango each tenant has its own copy of the tables, so the command is migrate_schemas --tenant instead — it applies the same migration to every tenant schema.

5 Django REST Framework (DRF)

DRF is the toolkit we use to build REST APIs on top of Django. The two key pieces:

View — a class with get(), post() etc. methods that handle HTTP requests for one URL. request.data gives you the JSON body as a dict; you return a response.

Serializer — turns model instances into dicts (for the response) and validates incoming dicts before saving. Think of it as a translator between Python objects and JSON.

6 The AST (Abstract Syntax Tree)

When Python reads your source code, it doesn't run it line-by-line as text. It first parses it into a tree of node objects: Import, Call, Attribute, Name, etc. This tree is the AST.

ast.parse(source) gives you the tree. ast.NodeVisitor is a helper class that walks the tree and calls visit_Import, visit_Name etc. as it encounters each node type. You override those methods to "look at" specific kinds of code.

7 Threads & thread-local

One Python process can run multiple "threads" concurrently — independent flows of execution sharing the same memory. A web server might use one thread per request; a celery worker uses one thread per task (by default).

threading.local() is a special object whose attributes are private to each thread. We use it to track "what execution is this thread currently running?" without passing the ID through every function call.

8 Context managers (with)

A with block guarantees cleanup. with open("x") as f: closes the file automatically when the block exits — even if an exception was raised.

We use @contextmanager to write our own: the function yields once, code inside the with runs, then control returns to the function for cleanup in a finally. The connection guard uses this to switch on the guard flag, run user code, switch off — even if user code crashes.

9 Closures & function factories

A function defined inside another function can capture variables from the enclosing scope. The inner function "closes over" them. So if you write a function that returns another function, the returned function can remember values from its birthplace.

We use this in the connection guard: _blocked_method_factory(original, name) returns a new function that remembers the original method and the name to use in error messages. Cleaner than building a lookup table.

10 Transactions & atomic()

A database transaction groups several statements into one all-or-nothing unit. Either every change inside commits together, or they all roll back.

Django's with transaction.atomic(): wraps a block. If the block exits normally → COMMIT. If it raises → ROLLBACK. Settings like SET LOCAL are scoped to the transaction too: they take effect inside the block and revert at the end automatically.

11 FileField & storage backends

Django's FileField stores a file path (string) in the database. The actual bytes go somewhere else — a "storage backend." In dev that's a local folder; in production it might be S3.

Switching backends is a settings change, not a code change. We use this to make our file rows portable: snippets and runs work the same whether the bytes live on disk or in S3.

12 The logging module

Python's import logging gives every part of an app a way to emit messages with a level (DEBUG / INFO / WARNING / ERROR / CRITICAL). You don't write print() in production code — you call logger.info(...).

A handler decides what to do with each message: write to console, send email, etc. We attach a custom handler that writes each message as a row in our log line table so they show up in the run's live tail.

13 ConnectionProxy & the real wrapper

django.db.connection is not a database connection — it's a proxy object. When you call connection.set_tenant(t), the proxy looks up the real connection (per-thread) and forwards the call.

To patch the set_tenant method permanently, we have to patch the real class behind the proxy, not the proxy itself. Hence the line cls = type(connections[DEFAULT_DB_ALIAS]) in the connection guard.

14 auditlog

Zango ships an auditlog library. Calling auditlog.register(MyModel) tells it: every time a row of this model is created, updated, or deleted, write an audit row describing the change and the user who made it.

We register all four core models. The high-volume log line table is deliberately NOT registered (auditing log lines would explode the audit table).

Key design decisions:

◇ UUID PKs everywhere

Avoids enumeration; safe to embed in URLs; no autoincrement collisions between runs.

◈ Two file scopes

CodeSnippetFile is what the editor manipulates; on run, each row is snapshotted into CodeExecFile(kind=input) with FK back to its source. Outputs only ever live at the execution level.

⌂ Source snapshot on every run

CodeExecution.source_snapshot is the literal code that ran. Edits to the snippet later don't rewrite the history.

◊ Hard delete on snippet files

No is_deleted flag. The source_snippet_file FK is SET_NULL so past executions retain their own row. Storage blobs are kept alive while any exec file still references them.

How a read works

1

Name sanitization

_sanitize_name strips path traversal (..), NUL bytes, leading dots; takes only the basename; caps at 255 chars.

2

DB lookup

CodeExecFile.objects.get(execution_id=<current>, kind="input", name=<sanitized>). CodexecFileNotFound if absent.

3

Open via storage

obj.file.open("rb") — Django's FileField resolves through STORAGES["default"] (S3 / local).

4

Return bytes / FileRef

Caller gets raw bytes (read), decoded text (read_text), parsed JSON (read_json), or a file-like (open).

How a write works

1

Sanitize + serialize

Same name sanitization. String → utf-8 bytes; objects → JSON via json.dumps(..., default=str).

2

Compute sha256 + size

Captured upfront for the model row.

3

Insert / replace CodeExecFile row

get_or_create(execution_id, kind="output", name=…). If overwriting, the old blob is delete(save=False)'d first.

4

Upload bytes

obj.file.save(name, ContentFile(payload), save=True) — Django writes through to S3 with the upload_to=_exec_file_upload_to callable, producing a UUID storage key.

5

Return FileRef

Thin proxy with .name / .size / .content_type / .sha256 for the snippet to inspect.

Filename preservation: CodeExecFile.name is the source of truth. The storage key uses a UUID, so two same-named files in different runs never collide. The download URL is generated with ResponseContentDisposition baked in so the browser saves with the original name.
07 · Celery executor

tasks.py · codexec_executor

The heart of the feature. Resolves tenant → loads workspace → AST recheck → wraps stdout/stderr → opens isolated transaction → installs ConnectionGuard → binds codexec helper → exec the source → captures result, stdout, stderr, traceback, duration. Catches exceptions inside the atomic so output files written before the failure still commit.

Pseudocode

pyapps/code_execution/tasks.py
@shared_task(bind=True, name="zango.code_execution.codexec_executor") def codexec_executor(self, execution_id, tenant_name): tenant = TenantModel.objects.get(name=tenant_name) connection.set_tenant(tenant) execution = CodeExecution.objects.select_related("snippet").get(pk=execution_id) execution.status = ExecStatus.RUNNING; execution.started_at = now(); execution.save(...) ws = Workspace(connection.tenant, as_systemuser=True); ws.ready() # L1 recheck — catches deny-list changes between save and pickup if validate(execution.source_snapshot): return _finalize_error(...) line_logger = LineLogger(str(execution.id)) stdout = CapturingStream(STDOUT_CAP_BYTES, "stdout", "out", line_logger) stderr = CapturingStream(STDERR_CAP_BYTES, "stderr", "err", line_logger) # Bridges so logging.* and loguru also land in the live tail stdlib_handler = _CodexecLogHandler(line_logger) stdlib_handler.addFilter(_NoisyLoggerFilter()) # drops botocore/urllib3/... stdlib_handler.setLevel(logging.INFO) root_logger.addHandler(stdlib_handler) loguru_sink_id = loguru.add(lambda m: line_logger.emit("info", m.record["message"]), ...) saved_stdout, saved_stderr = sys.stdout, sys.stderr sys.stdout, sys.stderr = stdout, stderr try: with transaction.atomic(): with connection.cursor() as c: begin_isolated_session(c, tenant.schema_name, execution.snippet.timeout_seconds) with install_connection_guard(): # L4 with codexec.bind_execution(str(execution.id)): try: code = compile(execution.source_snapshot, "<codexec>", "exec") exec(code, namespace) return_value = namespace.get("codexec_result") except SoftTimeLimitExceeded: status = ExecStatus.TIMEOUT; ...capture... except BaseException as e: status = ExecStatus.FAILED; ...capture... finally: stdout.flush(); stderr.flush(); line_logger.flush() sys.stdout, sys.stderr = saved_stdout, saved_stderr root_logger.removeHandler(stdlib_handler) loguru.remove(loguru_sink_id) execution.status = status execution.ended_at = now(); execution.duration_ms = ... execution.stdout = stdout.getvalue(); execution.stderr = stderr.getvalue() execution.return_value, execution.return_value_truncated = serialize_return(return_value) execution.save()

Why exceptions are caught inside the atomic

If we let the exception escape transaction.atomic(), Django would roll back every insert that happened inside — including the CodeExecFile rows the user already wrote with codexec.write(...). So the executor catches everything inside, records what went wrong on execution.exception_*, and lets the transaction commit cleanly. Output files survive a failed run.

08 · Isolation

L3 search_path + statement_timeout · L4 ConnectionGuard

L2 (per-tenant Postgres role) is parked. L3 + L4 are what's actually running.

L3 — begin_isolated_session

pyapps/code_execution/isolation.py
def begin_isolated_session(cursor, schema_name, timeout_seconds): schema = _validate_identifier(schema_name, "schema_name") timeout_ms = max(1000, int(timeout_seconds) * 1000) cursor.execute(f'SET LOCAL search_path = "{schema}", public') cursor.execute(f"SET LOCAL statement_timeout = {timeout_ms}")

SET LOCAL bounds the change to the current transaction. Any SET search_path the snippet issues is also LOCAL to the same transaction and reverts at commit. statement_timeout caps single queries at the snippet's wall-clock budget.

L4 — ConnectionGuard

Patches tenant-switching methods on the live DatabaseWrapper class with a thread-local flag. Outside an active guard, original methods run untouched. Inside, attempts to call them raise TenantSwitchBlocked.

pyapps/code_execution/guard.py
# Patched once, at startup of first guard activation. _GUARDED_METHODS = ("set_tenant", "set_schema", "set_schema_to_public", "_set_schema") def _install_patches_once(): cls = type(connections[DEFAULT_DB_ALIAS]) # real DatabaseWrapper, not the proxy if getattr(cls, "_codexec_guard_installed", False): return for name in _GUARDED_METHODS: if hasattr(cls, name): original = getattr(cls, name) setattr(cls, name, _blocked_method_factory(original, name)) cls.__setattr__ = _blocked_setattr_factory(cls.__setattr__) # for `.schema_name = …` cls._codexec_guard_installed = True @contextmanager def install_connection_guard(): _install_patches_once() prev = getattr(_guard_active, "active", False) _guard_active.active = True try: yield finally: _guard_active.active = prev

Because the patch lives on the class but only activates when the thread-local flag is set, normal application traffic (other tenant requests, async tasks, etc.) is unaffected — patched methods just call the original.

11 · Logging

How print, logging, and loguru all land in the live tail

Three independent capture mechanisms fan into a single LineLogger instance. Lines are batched (32 rows per flush) and written via bulk_create to CodeExecutionLogLine. The log-tail endpoint reads after a seq cursor; the UI polls every 1s while the run is non-terminal.

⌨ stdout / stderr

The executor wraps sys.stdout and sys.stderr with CapturingStream. Each newline flushes a line into LineLogger. Caps: 1 MB total stdout, 256 KB stderr, 8 KB per line.

🐍 stdlib logging

_CodexecLogHandler attached to the root logger. Filters out botocore / boto3 / urllib3 / s3transfer / asyncio / django.db.backends / PIL (otherwise the tail floods with library internals). Level INFO+.

🌠 loguru

A loguru sink added at executor entry that forwards each record's message to LineLogger with the matching level. Removed at finally.

Recursion safety

Without care, the bridges can loop: a flush hits a DB error, that error logs, the logger calls the handler, the handler inserts a log line, the insert errors, etc. — RecursionError. Two guards prevent it:

  • LineLogger._flushing — set to True for the duration of a bulk_create. The handler reads this and short-circuits.
  • Silent DB errors — flush() wraps bulk_create in a bare try/except: pass. Logging paths must never raise.
12 · File storage

Two-stage storage with reference-counted blobs

Same Django STORAGES["default"] backend the rest of Zango uses. For project newmi that's S3MediaStorage (django-storages); fallbacks gracefully to local FileSystemStorage.

Storage layout

<tenant>/codexec/snippets/<snippet_id>/<file_uuid>       ← CodeSnippetFile uploads
<tenant>/codexec/runs/<exec_id>/<kind>/<file_uuid>       ← CodeExecFile (inputs & outputs)

Filename preservation

Storage backends (especially S3) frequently rewrite filenames. We never trust the filename to round-trip — instead the DB column name is the source of truth, and the storage key uses a UUID. Downloads then bake the original name into S3's ResponseContentDisposition via a presigned URL:

Scheduled jobs

15 · Gaps

What's parked and what's deferred

⚠ L2 · per-tenant Postgres role parked

The original spec had a codexec_<schema> Postgres role per tenant with USAGE only on its own schema. Snippets would SET LOCAL ROLE to it so cross-schema queries fail with permission denied. Parked by product direction. Without it, fully-qualified raw SQL like SELECT * FROM "tntApp"."x" can still cross tenants. L1 + L3 + L4 cover the common paths but not this one.

The API supports filtering executions across all snippets (and CSV export). A dedicated UI page wasn't built — current flow (snippet row → execution detail → sidebar) covers most needs.

◔ Fine-grained policy enforcement deferred

Three policies are seeded (codexec.use / .edit / .run) but the API is gated only by IsAuthenticatedPlatformUser. Per-policy enforcement is a platform-wide pattern that needs separate work.

Phase 3 — not started

Per-tenant rate limits · approval workflow (draft → published) · diff view between versions · parameterized runs (typed inputs) · cron scheduling (overlap with AppTask) · saved snippet fixture files (auto-staged on every run) · websocket-based live tail (replacing polling).

src/zango/apps/code_execution/validator.py
427 lines · AST-based deny-list validator (L1 isolation)

This file is pure Python — no Django, no database, no file I/O. Its job is to read a snippet of user code and decide whether it contains any patterns we don't want to allow. Reading top-down: first there's a bunch of "deny lists" (sets of bad names, bad modules, bad attribute paths), then a couple of result types (Violation and CodeValidationError), then _DenyVisitor — the workhorse class that walks through the parsed source — then the two public functions validate() and assert_valid().

Background · The Abstract Syntax Tree (AST) When Python is about to run your code, it first parses the source text into a tree of node objects. Source like import os becomes a Module node containing an Import node containing an alias with name="os". Every node remembers its line number and column. Python's standard library exposes this tree via ast.parse(source). You can walk the tree without running the code — which is exactly what we want for a validator. Python ships a helper class, ast.NodeVisitor, that does the walking for you. You subclass it and override methods named visit_Import, visit_Call, etc. The visitor calls the right method as it encounters each node type. Any node type you don't care about, you don't override — the visitor just walks past it.

The deny-list constants

BANNED_IMPORT_PREFIXES = { "subprocess", "socket", "ctypes", "importlib", "pickle", "marshal", "multiprocessing", "pty", "resource", "_thread", "django_tenants.utils", } BANNED_FROM_IMPORT_NAMES = { "schema_context", "tenant_context", "get_tenant_model", "connections", "system", "popen", "execv", … } BANNED_NAMES = { "eval", "exec", "compile", "__import__", "globals", "locals", "vars", "setattr", "delattr", "breakpoint", "input", }

Sets (not lists) so membership checks are O(1). Six in total — one per "kind" of violation the visitor can detect. Keeping them at module level (not class attributes) means they're loaded once at import time, not per-validation.

Prefix vs exact match: BANNED_IMPORT_PREFIXES uses prefix matching ("subprocess" blocks "subprocess.run" too) because Python imports are dotted. The other sets are exact-name matches.

"connections" in BANNED_FROM_IMPORT_NAMES blocks from django.db import connections (the multi-DB dictionary). We don't want snippets sidestepping the connection guard by reaching for a different alias.

BANNED_DUNDERS = { "__class__", "__bases__", "__subclasses__", "__globals__", "__builtins__", "__import__", "__code__", "__dict__", "__mro__", "__getattribute__", "__reduce__", "__reduce_ex__", } BANNED_ATTR_PATHS = { ("connection", "set_tenant"), ("connection", "set_schema"), ("connection", "schema_name"), ("connection", "tenant"), ("django", "db", "connections"), ("os", "system"), ("os", "popen"), … } SQL_DANGER_PATTERNS = [ re.compile(r"\bSET\s+search_path\b", re.IGNORECASE), re.compile(r"\bSET\s+SCHEMA\b", re.IGNORECASE), re.compile(r"\bSET\s+ROLE\b", re.IGNORECASE), re.compile(r"\bSET\s+SESSION\s+AUTHORIZATION\b", re.IGNORECASE), ]

BANNED_DUNDERS blocks reflection escape hatches: ().__class__.__bases__[0].__subclasses__() is the classic sandbox escape, walking up to object then enumerating every class in the interpreter.

BANNED_ATTR_PATHS is a set of tuples representing attribute chains. The visitor walks chains like connection.cursor.set_tenant into ("connection", "cursor", "set_tenant") and prefix-matches against this set.

SQL_DANGER_PATTERNS are case-insensitive regex patterns that fire on string literals containing SET-style SQL that could change the session schema or role. Defense against cursor.execute("SET search_path TO 'evil'").

Violation & CodeValidationError

@dataclass(frozen=True) class Violation: line: int col: int code: str message: str def to_dict(self) -> dict: return { "line": self.line, "col": self.col, "code": self.code, "message": self.message, } class CodeValidationError(Exception): def __init__(self, violations): self.violations = violations super().__init__(f"{len(violations)} validation violation(s)")

@dataclass is a Python decorator that auto-generates the boring boilerplate of a simple data-holder class: __init__, __repr__, __eq__. Instead of writing def __init__(self, line, col, code, message): self.line = line; … you just list the fields with their types and the decorator does the rest. frozen=True makes the resulting object immutable — once created, you can't reassign its fields. Immutable objects are hashable in Python, which we'll use later to put violations in a set for deduplication.

code is a short label like "DENY_IMPORT", "DENY_NAME", or "DENY_DUNDER". The React frontend reads this and picks an icon and color for each violation kind.

to_dict() turns a Violation into a plain dict the API can hand to json.dumps. Python's built-in JSON encoder doesn't know how to serialize dataclass instances out of the box — an explicit to_dict() avoids any surprises in the response payload.

CodeValidationError exists for callers who prefer "raise on bad input" rather than "return a list of problems". None of the current views raise it — they all use the list-returning validate() directly — but it's there for future use.

_DenyVisitor — the AST walker

class _DenyVisitor(ast.NodeVisitor): def __init__(self): self.violations: List[Violation] = [] self.alias_to_module: dict = {}

Subclassing ast.NodeVisitor means we get the walking logic for free — we just say "when you encounter an Import node, run my visit_Import; when you encounter an attribute access, run my visit_Attribute". The Python standard library handles all the recursion through child nodes.

self.violations is the result we'll build up — every time a check fires, we append a Violation here.

The most subtle visitor. Triggered by any x.y expression. Two checks:

This file gives us our last line of defense against snippets trying to switch tenants at runtime. The basic idea is: take Django's database connection class and replace its tenant-switching methods with wrapper functions that check a flag before doing anything. When the flag is off, the wrapper just runs the original method like nothing happened. When the flag is on (which we turn on only while user snippet code is running), the wrapper refuses and raises an error.

Background · Monkey-patching a class "Monkey-patching" is the Python term for modifying a class after it's already defined. Because Python lets you reassign attributes on a class object at any time, you can literally write SomeClass.foo = my_new_function and now every instance.foo() call across the whole program runs your replacement. Use sparingly — it surprises future readers — but it's the cleanest answer when you need to inject behavior into a class you don't own (Django's DatabaseWrapper in our case).
Background · "Idempotent" An operation is idempotent if running it twice has the same effect as running it once. Our patch-install function checks "did I already install?" and returns early if so — so calling it from a hundred concurrent worker threads still results in exactly one patched class. Safe to call freely.

Module-level state

threading.local() is a special object from the standard library that gives you per-thread attribute storage. Normally, setting obj.foo = 1 is visible to every thread that sees obj. But a threading.local instance is different — each thread sees its own private copy of the attributes. So one thread can set _guard_active.active = True while another thread sees _guard_active.active as missing.

Why per-thread storage when celery runs one task at a time? With celery's default pool (prefork), each task runs in a separate child process, so there's no thread-sharing concern — a plain global variable would work. But celery has alternative pools:

  • -P threads --concurrency=N — N tasks run on N threads inside the same process. They share memory.
  • -P gevent — many tasks run as cooperative coroutines in one thread (threading.local is monkey-patched by gevent to be greenlet-local).

In those configurations, two snippets could be executing at the same time in the same process. A global flag would let one snippet's run accidentally turn the guard off for another's. threading.local sidesteps the issue entirely. In the default prefork setup it costs nothing (each process has one thread, so the local just acts like a global); under the threaded pools it's what makes the design correct. Defensive coding that's free in the simple case.

There's a second, subtler reason that applies even under prefork. The class we patch (Django's DatabaseWrapper) is shared by every database operation in the process — including the executor's own bookkeeping like execution.save(...) and the log line inserts. We need the guard "on" only while exec-ing user code, "off" for our own queries. That's a temporary flag, not a permanent state. The threading.local just makes the flag scoped correctly.

TenantSwitchBlocked is our custom exception class. It subclasses Python's built-in PermissionError so user code that catches PermissionError (a common pattern) will also catch ours — they're getting a permission error in a real sense, just one we invented.

_is_active() reads the per-thread flag. Notice getattr(_guard_active, "active", False) with a default — if a thread has never set the flag, the attribute simply doesn't exist, and accessing it directly would raise AttributeError. The getattr form with a default value sidesteps that cleanly.

Method factory — the closure trick

This pattern — a function that creates and returns another function — is called a closure factory in Python. The returned inner function captures variables from the outer function's scope (here, original and name). Those captured variables are "remembered" for the lifetime of the inner function, even after the outer function has returned.

Why do we need a factory? Because we want to patch several methods (set_tenant, set_schema, set_schema_to_public, etc.) and each replacement needs to know its own original implementation and its own name to put in the error message. If we wrote a single shared replacement function instead, it would have no way to refer back to "which original am I supposed to fall back to?"

_blocked.__name__ = name sets the __name__ attribute on the replacement function to match the original method's name. This is purely cosmetic — it makes Python's stack traces, debugger, and repr() show the right name when something goes wrong inside the patched method.

__wrapped__ = original is a Python convention used by debugging tools like functools.wraps: setting this attribute lets anyone introspect their way back to the function we wrapped, which is helpful in tests and debugging.

The body of the replacement is tiny: if the thread-local flag says "guard is active right now", raise; otherwise just call the original method. That's the entire Layer-4 mechanism in five lines of real logic.

Why patch the class instead of the instance? Because every request / celery task gets a different connection object, but they all share the same class. Why save the previous value with prev = getattr(...) and restore that, rather than just setting it to False on exit? Because someone might nest with blocks. Background · SET LOCAL vs SET in Postgres Both change a session setting. SET changes it until the connection closes (which can mean: leaks into the next request that reuses the connection). SET LOCAL changes it only inside the current transaction — at COMMIT or ROLLBACK, Postgres automatically reverts to what it was before. We always use SET LOCAL here so the next celery task on the same worker starts with a clean slate.
Background · Why validate identifiers ourselves? In SQL you can pass values as parameters (WHERE id = %s), and the driver escapes them safely. But identifiers — table names, schema names — can't be parameterized. We have to interpolate them as strings. To avoid SQL injection, we run the schema name through a strict regex first: only letters, digits, underscores; max 63 chars. If it fails, we raise — better than handing "x; DROP TABLE …" to Postgres.

Identifier validation

_IDENT_RE = re.compile(r"^[A-Za-z_][A-Za-z0-9_]{0,62}$") def _validate_identifier(value, label): if not isinstance(value, str) or not value: raise ValueError(f"{label} must be a non-empty string") if not _IDENT_RE.fullmatch(value): raise ValueError(f"Unsafe {label}: {value!r}") return value

When you write SQL in Python, you should pass values as parameters so the driver escapes them safely for you: cursor.execute("SELECT * FROM t WHERE id = %s", [user_id]). But this trick only works for values. Identifiers — table names, column names, schema names — can't be parameterized this way. Postgres needs them as literal tokens at parse time. So if we want to write SET search_path TO <schema> with a dynamic schema, we have to build the SQL by concatenating strings. And the moment you concatenate strings into SQL, you're one buggy input away from SQL injection.

The defense is to not trust the schema name. We run it through a strict regex that rejects anything that isn't a plain identifier. If the value is something weird like "valid"; DROP TABLE users; --, the regex fails and we raise before any SQL is built. Once it passes the check, we know it's safe to interpolate.

Reading the regex: ^[A-Za-z_] means the first character must be a letter or an underscore; [A-Za-z0-9_]{0,62} means zero to sixty-two more letters, digits, or underscores after that; $ means end-of-string. Total 1–63 characters, which is exactly Postgres's identifier length limit.

Why fullmatch instead of match? re.match anchors only at the start of the string — it would accept "valid; DROP TABLE x" because the first part matches. re.fullmatch requires the regex to consume the entire string, which is what we actually want.

Why allow both upper and lower case? Some tenant schema names we encountered in real usage are mixed-case (e.g. CRASystem7063). Postgres preserves case when the identifier is double-quoted in SQL — and we are double-quoting in the actual SET statement below — so mixed case is legal and safe.

SET LOCAL vs SET: LOCAL bounds the setting to the current transaction; without it, the setting persists for the session (and contaminates the next celery task on the same worker). Since the executor wraps the run in transaction.atomic(), LOCAL is the right scope.

Why two settings? search_path controls which schema unqualified identifiers resolve to — set to (tenant, public) means SELECT * FROM patient looks in the tenant first, then in shared public tables. Without this, the search_path django-tenants left would carry forward. statement_timeout caps any single query at the snippet's budget. A runaway loop in user code can't hold a connection forever.

Schema is double-quoted — required for mixed-case identifiers in Postgres. Without quotes, CRASystem7063 would be normalized to lowercase and the query would fail.

Background · The @shared_task decorator A function decorated with @shared_task is a Celery task — it can be enqueued for background processing. Calling it directly (codexec_executor(eid, t)) runs it synchronously, just like a normal function. Calling .apply_async(args=[eid, t]) serializes the args, pushes them to the broker, and a worker process picks them up and runs the function. The running process is what we call "the celery worker".
Background · Why redirect sys.stdout? When user code calls print("hi"), Python writes "hi\n" to sys.stdout — by default, the worker's terminal. We replace sys.stdout with a custom object before exec()-ing the user code, so prints go to our object instead. We save the original first and restore it in a finally block — even if user code crashes, the worker's stdout goes back to normal.
Background · The infinite-recursion trap We hook into Python's logging system so user code that calls logging.info(...) also writes to our log table. The trap: if writing to the table fails, Django logs the failure (via logging), which fires our handler again, which tries to write again, fails again, logs again — boom, stack overflow. The fix is a "re-entry flag" the handler checks before each write — if the flag says "I'm already inside a write", skip.

So we filter aggressively.

LineLogger — buffered batched writes

class LineLogger: BATCH_SIZE = 16 def __init__(self, execution_id): self.execution_id = execution_id self._seq = 0 self._pending: list[CodeExecutionLogLine] = [] self._flushing = False def emit(self, level, message): if not message: return if message.endswith("\\n"): message = message[:-1] if not message: return if len(message) > LINE_CAP_BYTES: message = message[:LINE_CAP_BYTES] + "… [line truncated]" self._seq += 1 self._pending.append(CodeExecutionLogLine( execution_id=self.execution_id, seq=self._seq, level=level, message=message, )) if len(self._pending) >= self.BATCH_SIZE: self.flush() def flush(self): if not self._pending: return self._flushing = True try: try: CodeExecutionLogLine.objects.bulk_create(self._pending) except Exception: pass finally: self._pending = [] finally: self._flushing = False

LineLogger is what bridges "user code wrote a line" and "that line shows up in the database log table". It buffers incoming lines and flushes them in batches.

Why batch instead of inserting each line individually? Because hitting the database is expensive — there's a network round-trip per INSERT. A snippet that prints a thousand lines would do a thousand round-trips if we naively inserted each. bulk_create sends one INSERT statement with sixteen rows in it. That's roughly sixteen times faster.

BATCH_SIZE = 16 is a deliberate trade-off. A larger batch is more efficient but means lines sit in memory longer before being saved (worse if the worker crashes mid-run). Sixteen feels small enough that visibility feels prompt and big enough that the database isn't hammered.

_seq is a counter we maintain ourselves. Every time we accept a new line, we increment it and assign that value to the row's seq field. Why our own counter instead of relying on the row's auto-generated primary key? Because the primary key is a global BigAutoField across all runs in the tenant — for a single run, the IDs aren't contiguous (other runs' rows interleave). The frontend's polling cursor needs per-run sequence numbers so it can ask "give me lines newer than seq 47 for this run", which the seq field makes trivial.

Stripping a trailing newline is just hygiene. The database column stores clean lines; the frontend renders one line per row and adds the visual break itself.

Per-line cap with truncation marker: nothing stops a snippet from calling print("X" * 10_000_000). Without a cap, that one line would bloat our row's message column to 10 MB. We slice it to 8 KB and append "… [line truncated]" so the user sees what happened.

self._flushing = True for the duration of the database call is the recursion guard we set up earlier. The stdlib log handler we attached checks this flag before writing — so if the bulk_create call somehow triggers more log messages (Django logs DB errors, for example), the handler sees the flag is set and skips, breaking the would-be infinite loop.

The bare except: pass around the actual bulk_create looks alarming, but it's deliberate. Code paths that exist to log things must never raise things — if they did, the user's perfectly-fine snippet would crash because of a transient logging issue. Better to silently drop a batch and let the run continue.

CapturingStream — the stdout/stderr replacement

class CapturingStream: def __init__(self, max_bytes, label, level, line_logger): self.max_bytes = max_bytes self.label = label self.level = level self.line_logger = line_logger self._chunks: list[str] = [] self._size = 0 self._truncated = False self._line_buf = "" def write(self, data): if isinstance(data, bytes): data = data.decode("utf-8", errors="replace") elif not isinstance(data, str): data = str(data) if self._truncated: return len(data) # Per-line cap (very long lines clipped with marker) if "\\n" not in data and len(data) > LINE_CAP_BYTES: data = data[:LINE_CAP_BYTES] + "… [line truncated]\\n" remaining = self.max_bytes - self._size if len(data) > remaining: if remaining > 0: self._chunks.append(data[:remaining]) self._size += remaining self._chunks.append(f"\\n--- {self.label} truncated …") self._truncated = True else: self._chunks.append(data) self._size += len(data) # Line-emit: split on newlines, emit complete lines self._line_buf += data while "\\n" in self._line_buf: line, self._line_buf = self._line_buf.split("\\n", 1) self.line_logger.emit(self.level, line) return len(data)

When user code calls print("hello"), Python writes the string to sys.stdout. We've replaced sys.stdout with one of these CapturingStream objects, so the print actually calls stream.write("hello\n"). For Python to accept the substitute, the substitute needs to look like a file — at minimum a write method, plus flush, writelines, and isatty. That's what's implemented here. There's no formal "interface" in Python, just the convention that anything with these methods can stand in for a file. This is called duck typing.

Decoding bytes safely. Most callers pass strings, but some libraries write raw bytes to stdout (e.g. when dumping binary content). We accept either: if we get bytes, we decode them as UTF-8 with errors="replace". The "replace" option means malformed bytes become the Unicode replacement character (a black diamond with a question mark) instead of raising. Logs should never crash because of weird input.

Truncation logic. Once self._truncated flips to True, every later write is a no-op (we pretend to have written all the bytes by returning len(data), so the caller doesn't realize and try to retry). The one-time write that crosses the cap writes whatever fits plus a small "--- truncated at X bytes ---" marker so the user knows the buffer was cut.

Two buffers, two destinations. self._chunks accumulates the entire stream so it can be saved on the execution.stdout column at the end of the run. self._line_buf accumulates characters until a newline arrives, at which point one complete line is handed off to the LineLogger (which in turn writes it to the per-line table). A single write call can contain multiple newlines (think a multi-line string), so we loop until no newline is left in the buffer.

What about a partial line without a newline? It just stays in _line_buf. Two things can free it: the next write completing the line with a newline, or the executor explicitly calling flush() at the end of the run.
