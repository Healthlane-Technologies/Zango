Section 01How the audit ran

Multi-agent workflow. Six investigators in parallel, each focused on one concern, returning structured findings. A synthesis agent then deduplicated and consolidated.

codexec
Does the stdout proxy compose correctly with the codexec executor, which also swaps sys.stdout for per-execution capture? Are codexec task names correctly skipped by _SKIP_PREFIXES?
stream-wrappers
Any other code in Zango that swaps streams or adds logging.StreamHandler instances — would it un-wrap our proxy or get double-routed?
boot-ordering
runserver vs wsgi vs asgi vs celery: does apps.ready() run before setup_logging() in wsgi.py? Does calling setup_logging() twice double-add sinks?
log-consumers
CloudWatch connector's regex and Logs Insights query — do they still match the modified log format with the new [schema:domain] prefix on the local console formatter?
admin-endpoints
Permission classes on all 6 new views; URL ordering (literals before catch-all); tenant-name injection into CloudWatch patterns; throttle scope sharing.
async-thread-safety
ContextVar behaviour under async views, sync_to_async boundaries, threaded gunicorn workers, Celery prefork pool semantics.

Section 02 · CriticalThree findings, two fixed, one refuted

Orphaned loguru sinks if setup_logging() fails FIXED
apps/shared/platform_logs/apps.py:22-28
Critical
What the audit said

logger.remove() runs at the start of setup_logging() before any sink is added. The bare except Exception: pass swallows partial failures — loguru can end up with zero sinks and silently drop everything. Workspace devs would see logs vanish with no diagnostic.

Fix

Surface the exception via stdlib logging (independent of loguru, so it works even when loguru is broken). The boot transcript now records the failure.

-        except Exception:
-            # Never fail app boot because of logging plumbing.
-            pass
+        except Exception:
+            _boot_logger.exception(
+                "platform_logs: setup_logging() failed during ready(); "
+                "loguru may have no active sink. Workspace logger calls "
+                "will be silently dropped until this is repaired."
+            )
_tokens dict collisions when task_id is None FIXED
apps/shared/platform_logs/celery_print_capture.py:32-77
Critical
What the audit said

If multiple concurrent tasks fire task_prerun with task_id=None (rare but possible with custom signal emission), they all collide on the same _tokens[None] key. Earlier bindings get overwritten without being reset — ContextVar leaks until the worker process exits.

Fix

Skip install entirely when task_id is falsy — no clean way to look the tokens back up at task_postrun, so a no-op is safer than a guaranteed leak. Same guard added to _restore.

+    # Without a task_id we have no way to look the tokens back up at
+    # task_postrun — and using a falsy sentinel as a dict key would let
+    # multiple concurrent anonymous tasks collide on the same slot,
+    # leaking ContextVar bindings. Skip the install in that case.
+    if not task_id:
+        return

Also added per-token try/finally in _restore so a failure resetting one stream still clears the other.

Partial bind leak on bind_stderr failure REFUTED
apps/shared/platform_logs/celery_print_capture.py:52-60
Critical
What the audit said

"If bind_stdout() succeeds but bind_stderr() raises, the stdout token is already stored but the exception is swallowed without resetting it."

Why this is already handled

The existing code stages _tokens[task_id] = (stdout_token, None) after the first bind and before the second. When the second bind raises and the outer except clause swallows the exception, task_postrun still fires for the task — _restore retrieves the staged pair and resets the stdout token (stderr is None and skipped). No leak.

The audit conflated this case with the genuine "worker dies between prerun and postrun" scenario, which on prefork is moot (process death takes ContextVar with it) but is real for solo/gevent pools. The middleware-side variant of the same concern was fixed by adding per-token try/finally for defence in depth.

_parse_tenants_param accepts arbitrary tenant names FIXED
api/platform/platform_logs/v1/views.py:425-427
Likely
What the audit said

Comma-separated tenant names from ?tenants=… were passed straight to CloudWatch filters without validation. No SQLi today, but a footgun — and lets clients probe arbitrary strings against the filter API.

Fix

Intersect requested tenants with TenantModel.objects.values_list('schema_name', flat=True); silently drop unknown names. Defence-in-depth on top of the connector-side escaping.

+def _known_tenant_schemas() -> set:
+    return set(TenantModel.objects.values_list("schema_name", flat=True))

def _parse_tenants_param(request):
    raw = request.GET.get("tenants", "") or ""
    requested = [s.strip() for s in raw.split(",") if s.strip()]
+    if not requested:
+        return None
+    known = _known_tenant_schemas()
+    out = [s for s in requested if s in known]
    return out or None

api/platform/platform_logs/v1/views.py:610-614

What the audit said

Middleware reset order didn't survive partial failures FIXED
apps/shared/platform_logs/middleware.py:39-40
Likely
What the audit said

If reset_stderr raises (and the exception path itself writes to stderr), stdout is still bound and misroutes those writes.

Fix

Each reset wrapped in its own try/finally so either failure still clears the other. Also handle the rare case where bind_stderr raises after bind_stdout succeeded — release the held stdout token before re-raising.

+        stdout_token = stdout_proxy.bind_stdout("zango.request.stdout")
+        try:
+            stderr_token = stdout_proxy.bind_stderr("zango.request.stderr")
+        except Exception:
+            stdout_proxy.reset_stdout(stdout_token)
+            raise
+        try:
+            return self.get_response(request)
+        finally:
+            try:
+                stdout_proxy.reset_stderr(stderr_token)
+            finally:
+                stdout_proxy.reset_stdout(stdout_token)
Stdlib logging output during codexec bypasses CapturingStream DEFERRED
apps/code_execution/tasks.py:406 + config/settings/base.py:546-549
Likely
What the audit said

Django's root StreamHandler caches a reference to sys.stderr at LOGGING-dict load time. When codexec swaps in CapturingStream, the cached reference still points at the old stream. So logging.info() from user code lands somewhere other than CodeExecution.stdout/stderr.

Why deferred

Pre-existing in main, not introduced by this PR — the asymmetry exists today on the feat/zango_logs baseline. The right fix (swap the handler's .stream attribute for the duration of execution, or remove+restore the console handler around the codexec block) belongs in a focused PR scoped to code_execution/tasks.py, not bundled into the logs follow-up.

Per-record connection.tenant lookup on the hot path DEFERRED
core/monitoring/utils.py:85-92
Likely
What the audit said

get_loguru_format touches django.db.connection.tenant on every loguru record. Under heavy print/log volume that's a repeated DB-state inspection on the logging hot path.

Why deferred

No measured production impact yet. connection.tenant is an in-memory attribute set by the tenant middleware — not a query — so the cost is one attribute access and a couple of getattr calls per record. Worth optimising if benchmarks show it matters, but premature without numbers. The eventual fix (cache (app_name, domain) in a ContextVar set once at request/task entry) needs the middleware to do the cache write, which is a coordinated change.

Section 06 · Confirmed safeWhat the audit looked at and cleared

Areas the investigators specifically checked and found no issues with — useful to know we don't need to revisit them.

  • Log-line format compatibility. Both the local console formatter ([app:domain] …) and the staging/prod verbose formatter ([app:domain][timestamp] …) are matched by _TENANT_RE, by CloudWatch's literal "[{app_name}:" substring filter, and by Logs Insights' re.escape pattern. No existing parser breaks.
  • Permissions on all 6 admin endpoints. Every Platform*View has permission_classes = (IsSuperAdminPlatformUser,). URL ordering puts literal paths (connectors/, components/) before the catch-all <str:component>/. Throttle scopes (platform_logs.browse, platform_logs.tail) are unique to platform endpoints — no quota collision with per-app calls.
  • Boot ordering. settings.ENV is available before apps.ready() fires. setup_logging() starts with logger.remove(), so wsgi/asgi calling it again after ready() is a no-op. Codexec tasks correctly hit the _SKIP_PREFIXES path, so the proxy stays in passthrough mode during codexec runs.
  • Single-tenant vs multi-tenant query flow. _apply_tenant_filter correctly pushes single-tenant down to CloudWatch and errors out on multi-tenant when the connector isn't on the verbose format.
