Platform Logs · Follow-ups

Close the gaps surfaced on staging
and add a platform-wide log viewer.

PR #574 shipped the connector + per-app Runtime Logs tab. Testing on staging revealed four real gaps in the producer side: stdlib loggers outside the django.* family stay file-only, loguru output bypasses our LOGGING dict entirely, third-party DEBUG floods CloudWatch, and Celery print() calls from workspace tasks land raw. This doc collects each fix, then adds a new platform-admin view that mirrors the Logs tab but without app scoping.

Overview

What this PR adds

Five pieces of work — four small producer-side patches that close the gaps we found while testing on staging, and one new platform-level view that lets platform admins see every tenant's lines in one place.

Producer fixes (settings + 1 hook)

  • Fix 1 · Add LOGGING["root"] so workspace and zango app loggers reach CloudWatch.
  • Fix 2 · Mute botocore / urllib3 / etc. DEBUG noise.
  • Fix 3 · Tenant-aware loguru format so every loguru call carries the [schema:domain] prefix.
  • Fix 4 · Celery task_prerun stdout/stderr wrapper so print() from workspace tasks becomes a logger call.

New feature

  • A new Platform → Logs page (sibling of Apps, User Management, Settings) that shows every tenant's lines.
  • Gated to platform superadmins.
  • Same connector pipeline; just no app_name filter applied at read time.
  • Adds a Tenant column so admins can correlate across schemas.
What we saw on staging

The four producer-side gaps

With PR #574 deployed and the staging override (LOGGING["loggers"]["django"]["handlers"] = ["file", "console"]), the Django request/security/template loggers showed up in CloudWatch with the verbose [schema:domain] prefix. But the test sample turned up four kinds of lines that didn't get the prefix:

Symptom Sample line in CloudWatch Why
Workspace logger.X() Policy 'OperationsAPIView' not found during auto-role attachment … Logger workspaces.… has no handlers; propagates to root; no root config → Python's lastResort stderr handler with %(message)s only. No prefix.
Third-party DEBUG DEBUG:urllib3.connectionpool:https://o116615… Same path as above. Adds a flood — bottocore/urllib3 emit hundreds of DEBUG lines per request.
Loguru 34|2026-06-02 11:13:20.556|INFO|__codexec__:<module>:7 - hello, codexec! Loguru is a separate ecosystem. core/monitoring/logging.py configures it with get_loguru_format from core/monitoring/utils.py. That format has no tenant prefix.
Celery print() Error while processsing Traceback (most recent call last): / r.status = res_text['results'][0]… print() writes directly to sys.stdout — bypasses Python's logging system entirely. No LOGGING dict change can intercept it.
Validation already in flight: the staging override at staging-deployment#1262 adds Fix 1 + Fix 2 as a quick check. This PR moves those settings into the platform code and adds Fix 3 + Fix 4 + the new Platform Logs view.
Fix 1 · Settings

Root logger catches everything not explicitly routed

Currently LOGGING["loggers"]["django"] is the only logger explicitly attached to the console handler. Every other logger in the process (workspace code, zango.apps.*, third-party libs) falls through to Python's lastResort handler with a bare %(message)s format. Adding a root entry redirects them all to the console handler — which already has the tenant_filter + verbose formatter.

Code change · backend/src/zango/config/settings/base.py (inside setup_settings)

pyafter the existing django logger configuration
# Catch every logger that doesn't have an explicit entry — # workspace code, zango.apps.*, third-party libs. Otherwise these # fall through to Python's lastResort handler which strips levels # and the tenant prefix. settings.LOGGING["root"] = { "handlers": ["file", "console"], "level": "INFO", }

Acceptance

  • A workspace logger.warning(...) call shows up in CloudWatch as [schema:domain][asctime] WARNING [path:func:line] message.
  • The Runtime Logs tab filters those lines correctly to the current app.
Fix 2 · Settings

Mute noisy third-party loggers

With root at INFO, third-party libs that emit DEBUG floods (botocore signing every S3 PUT, urllib3 every connection pool checkout) would drown CloudWatch. Raise their threshold to WARNING but leave them propagating — so genuine warnings still get the tenant prefix.

Code change · same file, right after Fix 1

pysettings/base.py
for noisy in ("botocore", "urllib3", "s3transfer", "boto3"): settings.LOGGING["loggers"][noisy] = { "level": "WARNING", "handlers": [], "propagate": True, }

handlers: [] + propagate: True means "no direct handlers, but my records still bubble up to root once they pass the WARNING threshold." Effectively a pre-filter.

Acceptance

  • No DEBUG:botocore.auth:StringToSign:… lines in CloudWatch.
  • Genuine botocore WARNINGs still appear, with the tenant prefix.
  • CloudWatch volume drops ~10× based on staging samples.
Fix 3 · Loguru

Tenant-aware loguru format

Loguru is a separate ecosystem — it doesn't use Python's stdlib logging machinery. So adding tenant_filter to console does nothing for loguru output. Zango already centralizes loguru config via core/monitoring/utils.py:get_loguru_format. The cleanest patch is to inject the tenant context directly into that format.

Code change · backend/src/zango/core/monitoring/utils.py

pyget_loguru_format
def get_loguru_format(record): file_name = _get_tenant_filename(record["name"]) # Tenant context — safe under unbound state (boot, mgmt cmds, # celery worker init before middleware binds a tenant). try: from django.db import connection tenant = getattr(connection, "tenant", None) app_name = getattr(tenant, "schema_name", "") if tenant else "" domain = getattr(tenant, "domain_url", "") if tenant else "" except Exception: app_name, domain = "", "" loguru_format = ( f"[{app_name}:{domain}]|" f"{os.getpid()}|" "<green>{time:YYYY-MM-DD HH:mm:ss.SSS}</green>|" "<level>{level}</level>|" f"<cyan>{file_name}</cyan>:" "<cyan>{function}</cyan>:" "<cyan>{line}</cyan> - " "<level>{message}</level>\n" ) return loguru_format

Why this and not "bridge loguru to stdlib"

Option A · Modify format recommended

~10 lines, surgical, doesn't touch existing rotation/OTel sinks. Loguru output stays in loguru-land but gets the prefix.

Option B · Bridge loguru → stdlib

Conceptually cleaner (one pipeline) but breaks the rotation sink, drops color codes, and the OTel exporter setup needs adjustment. Larger change.

Acceptance

  • Codexec snippet logger.info("hello, codexec!") emits as [tntApp:zel3-…]|34|… - hello, codexec! in CloudWatch.
  • Workspace tasks using loguru get the prefix.
  • log/zango.log on disk also gets the prefix (same format used by both sinks).
  • Mgmt commands run without a tenant: prefix is [:] (empty) but doesn't crash.
Fix 4 · Celery hook

print() from celery tasks → logger

Workspace tasks use print() liberally for ad-hoc debugging. Those writes go straight to sys.stdout — no logging hook can catch them. The pattern we used in the codexec executor works here too: swap stdout/stderr inside the task lifecycle with a stream that forwards lines to a logger.

Code change · new file apps/shared/platform_logs/celery_print_capture.py

pynew file
"""Swap sys.stdout/stderr inside a celery task scope so print() calls become logger.info()/error() calls — which means they pick up the verbose formatter + tenant_filter and end up in CloudWatch with the [schema:domain] prefix instead of as raw bare text.""" import logging import sys from celery.signals import task_prerun, task_postrun class _PrintStream: def __init__(self, original, logger_name, level): self._original = original self._log = logging.getLogger(logger_name) self._level = level self._buf = [] def write(self, s): self._original.write(s) # preserve normal stdout self._buf.append(s) if "\n" in s: full = "".join(self._buf); self._buf.clear() for line in full.splitlines(): if line: self._log.log(self._level, line) def flush(self): self._original.flush() _orig = {} # task_id → (orig_stdout, orig_stderr) @task_prerun.connect def _capture(task_id=None, **kw): _orig[task_id] = (sys.stdout, sys.stderr) sys.stdout = _PrintStream(sys.stdout, "celery.task.stdout", logging.INFO) sys.stderr = _PrintStream(sys.stderr, "celery.task.stderr", logging.ERROR) @task_postrun.connect def _restore(task_id=None, **kw): pair = _orig.pop(task_id, None) if pair: sys.stdout, sys.stderr = pair

Wire it from the same place we already wire celery_logging — add from . import celery_print_capture # noqa to apps/shared/platform_logs/apps.py:ready().

Acceptance

  • Workspace task doing print(traceback.format_exc()) now appears as proper ERROR log lines with the prefix.
  • The 5-line print() burst from ildspsp/reports/tasks.py:266 becomes 5 log records all tagged with [ildspsp:zel3-…].
  • Direct stdout writes (rare) still pass through unmodified — we preserve via self._original.write(s).
Fix 5 · Django middleware

print() from app views & helpers → logger

Same problem as Fix 4 — workspace devs print() in views, model methods, helpers — but the gunicorn web worker side needs a different mechanic because of threading. Fix 4's signal-based stdout swap is safe in celery (one task per worker process at a time); a sync gunicorn worker is the same shape (one request per process at a time), but a thread-based or async worker handles multiple concurrent requests. A naive sys.stdout swap from one thread would intercept writes from every other thread.

Approach · request-bound contextvar + a thread-aware stdout proxy

Install a process-wide sys.stdout / sys.stderr proxy once at startup. The proxy reads a contextvars.ContextVar on every write() — if a request is active, forward to the per-request logger; if not, fall through to the real stdout/stderr. The middleware just sets and clears the ContextVar around the request. No per-thread global swap.

Code change · new file apps/shared/platform_logs/stdout_proxy.py

pynew file — installed once at module import
import logging import sys from contextvars import ContextVar # Set by middleware (Fix 5b) and the celery task hook (Fix 4) so the # proxy knows which logger to forward to for the current "scope". # When None, writes pass straight through to the real stdout/stderr. _active: ContextVar = ContextVar("platform_logs_stdout_target", default=None) class _Proxy: def __init__(self, fallback, default_level): self._fallback = fallback self._default_level = default_level self._buf = [] def write(self, s): target = _active.get() if target is None: return self._fallback.write(s) # Pass through too — preserves normal terminal/CloudWatch visibility # at the raw stream level (so dev tooling that follows stdout # still sees the output). self._fallback.write(s) self._buf.append(s) if "\n" in s: full = "".join(self._buf); self._buf.clear() log = logging.getLogger(target) for line in full.splitlines(): if line: log.log(self._default_level, line) return len(s) def flush(self): self._fallback.flush() def isatty(self): return self._fallback.isatty() def install(): """Wrap sys.stdout/sys.stderr once. Safe to call from app.ready().""" if not isinstance(sys.stdout, _Proxy): sys.stdout = _Proxy(sys.stdout, logging.INFO) if not isinstance(sys.stderr, _Proxy): sys.stderr = _Proxy(sys.stderr, logging.ERROR) def bind(logger_name: str): """Returns a token suitable for `_active.reset(token)` after the scope ends.""" return _active.set(logger_name) def reset(token): _active.reset(token)

Wire-up

  1. Install the proxy once from apps/shared/platform_logs/apps.py:ready() — calls stdout_proxy.install().
  2. App side — Django middleware that calls stdout_proxy.bind("zango.request.stdout") at request start, stdout_proxy.reset(token) in finally.
  3. Celery side — Fix 4's task_prerun handler does the same: stdout_proxy.bind("celery.task.stdout") at start, reset at task_postrun. This replaces Fix 4's _PrintStream entirely — single primitive for both sides.

Why this is thread-safe

  • contextvars.ContextVar is per-execution-context (per-thread and per-async-task). Each request running concurrently has its own value.
  • The proxy reads the ContextVar lazily on every write(). A thread without a binding falls through to the real fallback stream untouched.
  • sys.stdout swap happens once at process start — no concurrent mutation, no race.

Code change · new file apps/shared/platform_logs/middleware.py

py
from zango.apps.shared.platform_logs import stdout_proxy class RequestPrintCaptureMiddleware: def __init__(self, get_response): self.get_response = get_response def __call__(self, request): token = stdout_proxy.bind("zango.request.stdout") try: return self.get_response(request) finally: stdout_proxy.reset(token)

Add to MIDDLEWARE in settings/base.py

py
MIDDLEWARE = [ # at or near the top — before any middleware whose own # print()/log calls we want captured. After SecurityMiddleware is fine. "zango.apps.shared.platform_logs.middleware.RequestPrintCaptureMiddleware", ... ]

Acceptance

  • A workspace view doing print("debug:", thing) appears in CloudWatch as a proper INFO line with the [schema:domain] prefix.
  • Two concurrent requests on a thread-based gunicorn worker don't cross-pollute — each gets its own ContextVar binding.
  • Boot-time / mgmt-command print() calls (no ContextVar set) still go to real stdout unmodified.
  • The "preserve normal stdout" path in _Proxy.write means tail -f /proc/<gunicorn-pid>/fd/1 in the container still shows the raw print, in addition to the formatted log line.
Fix 4 collapses into this. Once the proxy + ContextVar primitive exists, the celery task_prerun hook just calls the same bind / reset. Drop the separate _PrintStream class. One implementation, two scopes (request and task).
New feature

Platform → Logs page

The per-app Runtime Logs tab is good for an app developer poking at their own app's behavior. For a platform admin investigating an incident across multiple tenants, they currently need to either open every app or go to CloudWatch. A top-level Platform → Logs page (sibling of Apps / User Management / Settings) uses the same connector pipeline but drops the per-app filter and adds a Tenant column.

API

Endpoints

Mirror the in-app endpoints but at /api/v1/platform/logs/ with the admin permission class and no app_name injection.

Endpoint Purpose
GET /api/v1/platform/logs/components/List configured components for the current env (drives the picker).
GET /api/v1/platform/logs/<component>/Browse — latest N in window, no tenant filter.
GET /api/v1/platform/logs/<component>/tail/Live tail forward cursor, no tenant filter.
GET /api/v1/platform/logs/<component>/streams/List sub-streams for the picker.
GET /api/v1/platform/logs/<component>/facets/Cached levels + tenant set parsed from recent lines.
GET /api/v1/platform/logs/<component>/tenants/Optional · distinct tenants seen in the last 24h (drives the Tenant dropdown).

Query parameters

All the same as the in-app endpoints plus one addition:

  • &tenants=newmicare,tntApp,ildspsp — when present, the connector restricts to lines whose app_name matches one of the listed schemas. When absent, no tenant restriction.

Reuses the existing CloudWatch connector. The only change is the view layer: instead of filters.app_name = _tenant_schema_for(app_uuid), it passes filters.app_name = None (or honors the tenants list if present and translates it into CloudWatch filter pattern ORs).

Open

Decisions to make before coding

  • Root logger INFO vs WARNING as the default level. INFO surfaces useful zango app lines but adds volume; WARNING is quieter but hides "patient saved" / "task scheduled" kinds of breadcrumbs. Recommend INFO with the noise muters as a starting point.
  • Loguru fix path — confirm Option A (modify format) over Option B (bridge to stdlib). Recommend A.
  • Celery print capture · should it apply to the codexec executor too, or stay turned off for codexec tasks since the executor already has its own CapturingStream? Recommend: skip when task name starts with zango.code_execution..
  • Platform Logs · Tenant column visible by default, or hidden by default and shown only when a tenant filter is active? Recommend: visible always, hideable from the View menu.
  • Multi-tenant filter pattern — how many [name: ORs can CloudWatch handle in a single FilterLogEvents pattern? AWS doc says ~1024 chars per pattern. May need to chunk for >30 selected tenants. Worth confirming.
  • The new Platform Logs page navigation — do we want it inside the existing Settings shell (as a fourth tab next to Log Connectors), or as a separate top-level nav item? Recommend separate top-level — viewing logs is a different mental task from configuring connectors.
