Why this PR exists

PR #574 (the previous one) shipped Platform Logs: a way for tenant admins to see their app's CloudWatch logs inside the Zango UI. The whole feature relies on every log line being prefixed with [schema:domain] — that's how the connector knows which tenant a line belongs to.

During staging testing four leaks were found where lines reached CloudWatch without that prefix:

  1. Loggers like workspace and zango.apps.* weren't wired up to the formatter that adds the prefix.
  2. Loguru (a different logging library Zango uses alongside stdlib logging) had its own format that didn't include the prefix.
  3. Third-party libraries (botocore, urllib3) were flooding the logs with DEBUG noise.
  4. print() calls bypass Python's logging system entirely — no formatter, no prefix.

The other half of the PR adds a Platform → Logs page where super-admins can see logs across every tenant on one screen, with a Tenant filter and a Tenant column.

Change 1 · Pin the console handler at INFO

config/settings/base.py · line 286modified
"console": {
    "class": "logging.StreamHandler",

    # once other loggers route here.
    "level": "INFO",
    "formatter": "console",

Plain English

A Python "handler" is the thing that decides where a log line goes (file, console, network). Without a level, a handler accepts every record handed to it. Once we wired the django logger (which is at DEBUG) to also route here, every django.utils.autoreload.tick DEBUG record flooded the screen — hundreds of file-watcher records per second.

Pinning the handler at INFO caps what reaches stdout no matter which logger feeds it. Loggers can still set their own threshold lower; this just acts as a final gate.

Change 3 · Route the django logger to console + add a root logger

config/settings/base.py · line 521modified
settings.LOGGING["loggers"]["django"] = {
    # `console` ships to stdout → ECS awslogs driver → CloudWatch.
    # `file` keeps the on-disk copy. `propagate: False` prevents
    # double-writes via the root logger below.
    "handlers": ["file", "console"],
    "level": "DEBUG",
    "propagate": False,
}

# Catch every logger that doesn't have an explicit entry —
# workspace code, zango.apps.*, third-party libs. Otherwise these
# fall through to Python's `lastResort` handler which strips levels
# and the tenant prefix.
settings.LOGGING["root"] = {
    "handlers": ["file", "console"],
    "level": "INFO",
}
Plain English

The django logger. Before: routed only to the rotating file handler. Lines from Django (request log, autoreload, etc.) never reached the console handler — so CloudWatch never saw them with the tenant prefix. Now: routed to both.

The root logger. Python's logging uses a tree. Loggers without their own configuration "fall up" to their parent, all the way to root. Without a root configured, Python uses a built-in fallback that ignores formatters and filters — every workspace logger.warning(...) ended up at CloudWatch with no prefix. Adding an explicit root entry routes those records through our formatter.

propagate: False on django. Without this, every django record would be processed twice — once via the django logger's handlers and once via root. Setting it to False says "don't bubble up; my handlers already wrote you to file and console."

Plain English

boto3 and friends are extremely chatty at DEBUG and INFO — they log every AWS API call, every retry, every signature step. With the root logger now catching them at INFO, they'd flood CloudWatch with thousands of useless lines per minute.

Backend · 2 of 8core/monitoring/utils.py

This file builds the format string loguru uses for every line. We added the tenant prefix.

core/monitoring/utils.py · get_loguru_formatmodified
def get_loguru_format(record):
    file_name = _get_tenant_filename(record["name"])

    # Tenant context — safe under unbound state (boot, mgmt commands,
    # celery worker init before middleware/signals bind a tenant). The
    # Platform Logs CloudWatch connector uses this `[schema:domain]`
    # prefix to filter lines per-tenant.
    try:
        from django.db import connection

        tenant = getattr(connection, "tenant", None)
        app_name = getattr(tenant, "schema_name", "") if tenant else ""
        domain = getattr(tenant, "domain_url", "") if tenant else ""
    except Exception:
        app_name, domain = "", ""

    loguru_format = (
        f"<blue>[{app_name}:{domain}]</blue>|"
        f"<magenta>{os.getpid()}|</magenta>"
        "<green>{time:YYYY-MM-DD HH:mm:ss.SSS}</green>|"
        # ... level, file, function, line, message ...
    )
    return loguru_format
Plain English

Why try/except? This function runs on every loguru record, including ones emitted very early — during settings load, during management commands, during celery worker boot — when django.db.connection.tenant doesn't exist yet. Any error here would crash the log emit, which would crash the thing that was trying to log. The bare except trades fidelity (we get empty strings) for never crashing.

schema_name and domain_url. Two attributes django-tenants adds to the tenant object. The schema name is the database schema (e.g., newmicare); the domain URL is the host the tenant answers on (e.g., newmicare.zango.dev).

Loguru's <blue>...</blue> tags. Loguru renders ANSI color codes when the sink is a terminal. We wrap the prefix in <blue> so it stands out locally; CloudWatch strips the ANSI on display.

Backend · 3 of 8platform_logs/stdout_proxy.py (new)

The trickiest file in the PR. Workspace developers use print() a lot — those calls bypass the logging system entirely, so they reach CloudWatch as raw bytes with no level, no timestamp, no tenant prefix. This module sits between print() and the actual stream, turning every print into a loguru log record so it ends up shape-identical to a deliberate logger.info() call.

Why loguru, not stdlib logging?

Zango's project convention is loguru — workspace code is told to use from loguru import logger. Routing captured prints through loguru means a stray print("...") looks identical in CloudWatch to an intentional logger.info("..."): same tenant prefix, same format, same file/function/line columns. Routing through stdlib would create two visually-different shapes for the same kind of output.

Imports and ContextVars

stdout_proxy.py · top of filenew
import sys
from contextvars import ContextVar
from typing import Optional

from loguru import logger as _loguru_logger

_active_stdout: ContextVar[Optional[str]] = ContextVar(
    "platform_logs_stdout_target", default=None
)
_active_stderr: ContextVar[Optional[str]] = ContextVar(
    "platform_logs_stderr_target", default=None
)

_emitting: ContextVar[bool] = ContextVar(
    "platform_logs_stdout_emitting", default=False
)

_LOGGING_PACKAGES = ("loguru", "logging")

def _called_from_logging() -> bool:
    """True if any of the closest frames live inside a logging package."""
    frame = sys._getframe(2)
    for _ in range(8):
        if frame is None:
            return False
        mod = frame.f_globals.get("__name__", "")
        if mod.startswith(_LOGGING_PACKAGES):
            return True
        frame = frame.f_back
    return False
Plain English

What's a ContextVar? A variable whose value is scoped to the current "execution context" — roughly, the current thread or async task. If thread A sets a ContextVar to "hello", thread B reading the same ContextVar sees the default. This matters because Django serves requests on different threads or different async tasks; we need each request to have its own binding.

Why two for stdout/stderr? An earlier version of this code used one shared ContextVar, and binding stdout then stderr clobbered the first binding — every stdout write was logged under the stderr name. The fix is one var per stream.

Why _emitting? The first of two reentrancy guards. Covers the case where we initiated the loguru call: proxy → loguru → sink → stream → proxy. Setting _emitting=True before calling loguru.log makes the second proxy entry a passthrough.

Why _called_from_loguru()? The second guard, covering the case where loguru is called directly by workspace code while a binding is active: logger.error("foo") → loguru → sink → stream → proxy. Here _emitting is False (we never started this emission), so the proxy would forward loguru's formatted output back into loguru — tripping loguru's lock and triggering "Could not acquire internal lock because it was already in use". The frame check spots that the immediate caller of write() is inside the loguru package and passes through instead of forwarding.

Why frame inspection at all? Loguru doesn't expose a "currently emitting" flag we can read. The cheapest reliable signal is the call stack: when loguru's sink calls sys.stderr.write(formatted_text), there will always be a loguru frame two or three steps up. Walking eight frames is bounded and adds microseconds per write.

The proxy class

stdout_proxy.py · class _Proxynew
class _Proxy:
    def __init__(self, fallback, default_level_name, active_var):
        self._fallback = fallback
        self._default_level_name = default_level_name  # "INFO" / "ERROR"
        self._active_var = active_var
        self._buf: list[str] = []

    def write(self, s):
        if _emitting.get():
            try:
                return self._fallback.write(s)
            except Exception:
                return len(s)

        target = self._active_var.get()
        if target is None or not isinstance(s, str) or not s:
            try:
                return self._fallback.write(s)
            except Exception:
                return len(s)

        if _called_from_loguru():
            try:
                return self._fallback.write(s)
            except Exception:
                return len(s)

        self._buf.append(s)
        if "\n" in s:
            full = "".join(self._buf)
            self._buf.clear()
            token = _emitting.set(True)
            try:
                emit = _loguru_logger.bind(source=target).opt(depth=1)
                for line in full.splitlines():
                    if line:
                        emit.log(self._default_level_name, line)
            finally:
                _emitting.reset(token)
        return len(s)
Plain English, line by line

__init__ stores three things: the real stream (so we can pass writes through when nothing is bound), the loguru level name to use for forwarded records ("INFO" for stdout, "ERROR" for stderr), and which ContextVar to read.

if _emitting.get() — the reentrancy guard. If we're already inside a forwarded emission, just write to the real stream and stop. This is the loop-breaker.

target = self._active_var.get() — read the current binding for this stream. If no scope has bound anything, target is None.

Passthrough fast path. If there's no binding, or the input isn't a string, or it's empty, just write to the real stream. This is what happens during boot, in management commands, in shell sessions — anywhere a request middleware or celery hook hasn't bound a source.

self._buf.append(s) — buffer the write. print("hello") actually makes two calls: write("hello") and write("\n"). We can't emit a log line on the first call because we don't have a complete line yet.

if "\n" in s — once we see a newline, join the buffer, clear it, then forward each complete line as a loguru record.

_loguru_logger.bind(source=target) — stashes the stream label ("zango.request.stdout", "celery.task.stdout", …) into the record's extra dict so format strings can surface it if needed.

.opt(depth=1) — shifts loguru's reported caller frame one step up the stack, past _Proxy.write. Without this, every captured line's file/function/line columns would point at the proxy itself; with it, they point at the real print() site in the workspace code.

emit.log(self._default_level_name, line) — the actual emission. Loguru takes the level by name ("INFO" / "ERROR"). The record now has: tenant prefix (via get_loguru_format), correct caller frame (via opt depth), source extra (via bind), level (INFO for stdout, ERROR for stderr).

token = _emitting.set(True) / finally: _emitting.reset(token) — set the guard before forwarding, reset it after. set() returns a token; reset(token) restores the previous value. Wrapping in try/finally guarantees the guard gets cleared even if loguru blows up.

install() and the bind/reset helpers

stdout_proxy.py · module functionsnew
def install() -> None:
    if not isinstance(sys.stdout, _Proxy):
        sys.stdout = _Proxy(sys.stdout, "INFO", _active_stdout)
    if not isinstance(sys.stderr, _Proxy):
        sys.stderr = _Proxy(sys.stderr, "ERROR", _active_stderr)

def bind_stdout(logger_name): return _active_stdout.set(logger_name)
def bind_stderr(logger_name): return _active_stderr.set(logger_name)

def reset_stdout(token):
    try: _active_stdout.reset(token)
    except (LookupError, ValueError): pass

def reset_stderr(token):
    try: _active_stderr.reset(token)
    except (LookupError, ValueError): pass
Plain English

install() replaces sys.stdout and sys.stderr with our proxies. The isinstance check makes it idempotent — calling it twice is harmless. Idempotency matters because multiple boot paths (wsgi.py, asgi.py, runserver, celery) might all call it.

bind_stdout / bind_stderr. Set the target logger for the current context. They return a token the caller must hold onto and pass to reset_* later.

except (LookupError, ValueError) on reset. Two failure modes: the token might already have been consumed (LookupError), or it might belong to a different context (ValueError) if the request hops threads. Swallowing both is safe because in either case the binding is already gone or doesn't apply here.

Backend · 4 of 8platform_logs/middleware.py (new)

A Django middleware that binds the stdout proxy at the start of every HTTP request and unbinds it at the end.

middleware.py · entire filenew
from zango.apps.shared.platform_logs import stdout_proxy

class RequestPrintCaptureMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        stdout_token = stdout_proxy.bind_stdout("zango.request.stdout")
        stderr_token = stdout_proxy.bind_stderr("zango.request.stderr")
        try:
            return self.get_response(request)
        finally:
            stdout_proxy.reset_stderr(stderr_token)
            stdout_proxy.reset_stdout(stdout_token)
Plain English

Django middleware shape. __init__ runs once at server boot; __call__ runs for every request. get_response is the next middleware or the view itself. Whatever __call__ returns becomes the HTTP response.

Binding before get_response. We capture from this point on. Any print() in a view, signal handler, or helper called during the request gets forwarded to the zango.request.stdout logger.

try/finally. Even if the view raises, finally guarantees we unbind. Without that, a binding could leak across requests — the next request on the same thread would still be capturing under the previous request's tenant.

Reset order. Reset stderr first, then stdout — same order as bind, in reverse. Doesn't actually matter here because they're independent ContextVars, but the symmetry is conventional.

Backend · 5 of 8platform_logs/celery_print_capture.py (new)

The Celery equivalent of the middleware. Binds the proxy at task start, unbinds at task end. Uses Celery's signal system.

celery_print_capture.py · key bitsnew
from celery.signals import task_postrun, task_prerun
from zango.apps.shared.platform_logs import stdout_proxy

_SKIP_PREFIXES = ("zango.code_execution.",)
_tokens: dict = {}

def _should_skip(task_name):
    return bool(task_name) and any(task_name.startswith(p) for p in _SKIP_PREFIXES)

@task_prerun.connect
def _capture(task_id=None, task=None, **kwargs):
    name = getattr(task, "name", "") or ""
    if _should_skip(name):
        return
    stdout_token = None
    stderr_token = None
    try:
        stdout_token = stdout_proxy.bind_stdout("celery.task.stdout")
        _tokens[task_id] = (stdout_token, None)
        stderr_token = stdout_proxy.bind_stderr("celery.task.stderr")
        _tokens[task_id] = (stdout_token, stderr_token)
    except Exception:
        logger.exception("...")

@task_postrun.connect
def _restore(task_id=None, **kwargs):
    pair = _tokens.pop(task_id, None)
    if pair is None:
        return
    stdout_token, stderr_token = pair
    try:
        if stderr_token is not None:
            stdout_proxy.reset_stderr(stderr_token)
        if stdout_token is not None:
            stdout_proxy.reset_stdout(stdout_token)
    except Exception:
        logger.exception("...")
Plain English

Celery signals. Celery emits events at well-defined points in a task's lifecycle. task_prerun fires right before the task function runs; task_postrun fires right after. We hook both to bind/unbind the proxy.

_SKIP_PREFIXES. The codexec executor (a separate feature that runs user-supplied Python in a sandbox) already swaps stdout for its own per-execution capture. If our proxy also wrapped its stdout, the executor's capture would see our proxy instead of the real stream, and lines would get logged twice. Skipping codexec tasks keeps that executor as the sole owner of its own stdout.

_tokens dict. Maps each running task's ID to the pair of tokens we got from binding. task_postrun looks it up to know what to reset. Per-process dict — separate Celery worker processes don't share state, which is correct.

Staged assignment to _tokens. First we bind stdout and immediately record (stdout_token, None). Then we bind stderr and update to (stdout_token, stderr_token). If the second bind raises, we still have the first token recorded — _restore can clean it up. Without staging, a half-bound task would leak a stdout token forever.

Why catch Exception at top level? A bug in our logging plumbing should never crash a workspace task. Logging the exception (via logger.exception) records it for us to diagnose without aborting the work.

Helpers — parsing the tenants query param

views.py · helpersnew
import re

_TENANT_RE = re.compile(r"\[([^:\]]+):[^\]]*\]")
_MAX_FETCH_LIMIT = 500

def _parse_tenants_param(request):
    raw = request.GET.get("tenants", "") or ""
    out = [s.strip() for s in raw.split(",") if s.strip()]
    return out or None

def _apply_tenant_filter(filters, tenants, cfg):
    if not tenants:
        return filters
    if len(tenants) == 1:
        filters.app_name = tenants[0]
        return filters
    fmt = getattr(cfg, "fmt", "verbose")
    if fmt != "verbose":
        raise ConnectorError("Multi-tenant filtering requires verbose fmt...")
    filters.limit = _MAX_FETCH_LIMIT
    return filters

def _post_filter_tenants(log_page, tenants):
    if not tenants or len(tenants) < 2:
        return log_page
    wanted = set(tenants)
    keep = []
    for ln in log_page.lines:
        m = _TENANT_RE.match(ln.message or "")
        if m and m.group(1) in wanted:
            keep.append(ln)
    log_page.lines = keep
    return log_page
Plain English

The regex \[([^:\]]+):[^\]]*\]. Matches text shaped like [schemaname:domain] at the start of a log line. ([^:\]]+) captures the schema name (everything that isn't : or ]). The schema is what we use to decide "is this line from a selected tenant?"

_parse_tenants_param. Reads ?tenants=a,b,c from the URL, splits on commas, strips whitespace, drops empties. Returns None if nothing valid was passed so callers can short-circuit.

Single vs multi-tenant in _apply_tenant_filter. If exactly one tenant is requested, we push the filter all the way down to the CloudWatch query — the connector adds a pattern that only matches lines starting with [that-schema:. That's efficient: CloudWatch does the work, we get back exactly the lines we want.

Multi-tenant is harder. CloudWatch's filter pattern syntax can't express "match prefix A OR prefix B OR prefix C." So when more than one tenant is selected, we fetch everything in the time window (capped at 500) and filter on the server side after the fact. The fmt check refuses this on connectors that use plain or JSON formats — without the [schema:domain] prefix in the message, there's nothing to match and the UI would silently look empty.

_post_filter_tenants. Walks the fetched page, regex-matches the prefix of each line, keeps only those whose schema is in the requested set. The cursor is not adjusted — it still points at the end of the unfiltered page, so the next poll resumes at the correct place in CloudWatch's timeline.

const buildQs = useCallback((extra = {}) => {
  const p = new URLSearchParams();
  if (filters?.since) p.set('since', filters.since);
  if (filters?.until) p.set('until', filters.until);
  if (filters?.levels?.length) p.set('levels', filters.levels.join(','));
  if (filters?.streams?.length) p.set('streams', filters.streams.join(','));
  if (filters?.tenants?.length) p.set('tenants', filters.tenants.join(','));
  if (filters?.q) p.set('q', filters.q);
  p.set('limit', String(extra.limit || 200));
  Object.entries(extra).forEach(([k, v]) => {
    if (k === 'limit') return;
    if (v !== undefined && v !== null && v !== '') p.set(k, String(v));
  });
  return p.toString();
}, [filters]);
Plain English

URLSearchParams handles URL encoding for us — special characters in tenant names or search queries get escaped correctly.

The new line vs the in-app variant is p.set('tenants', ...). Every other field exists on both sides.

useCallback wraps the function so React only rebuilds it when filters changes, preventing downstream effects from re-running on every render.

Live tail polling

useLogTail.js · pollOncenew
const pollOnce = useCallback(async () => {
  if (!enabled || !component || pausedRef.current || !tailCursorRef.current)
    return;
  if (typeof document !== 'undefined' && document.hidden) return;

  const qs = buildQs({ after: tailCursorRef.current });
  const { response, success } = await triggerApi({
    url: `/api/v1/platform/logs/${component}/tail/?${qs}`,
  });
  if (pausedRef.current) return;
  if (!success) return;

  const fresh = response.lines || [];
  if (response.next_cursor?.token) {
    tailCursorRef.current = response.next_cursor.token;
  }
  if (fresh.length > 0) {
    setLines((prev) => {
      const tagged = fresh.map((l) => ({ ...l, _fresh: true }));
      tagged.reverse();
      return [...tagged, ...prev].slice(0, maxBufferedLines);
    });
  }
}, [/* deps */]);
Plain English

Three early returns at the top. Don't poll if we're paused, no component picked, no cursor (haven't done initial browse yet), or the browser tab is hidden — no point burning CloudWatch quota while the user isn't watching.

pausedRef.current instead of paused. A ref is read at call time, so even a "leaked" interval (e.g., from hot-module reload during dev) sees the current value. Reading the state variable directly would close over a stale value.

Double-check after fetch. The user might have paused while the request was in flight. If so, discard the response — don't advance the cursor, don't append lines. Prevents a paused view from suddenly displaying new lines on the next user action.

_fresh: true. A flag the table uses to briefly highlight newly arrived lines with a fade animation, so the user notices new content.

tagged.reverse(). The API returns lines oldest-first; we store newest-on-top, so we reverse before prepending.

.slice(0, maxBufferedLines). Cap memory at the buffer size (4000 by default). Old lines fall off the bottom.

const TENANT_RE = /^\[([^:\]]+):([^\]]*)\]\s*/;

function parseTenantPrefix(message) {
  if (!message) return { schema: null, rest: message || '' };
  const m = TENANT_RE.exec(message);
  if (!m) return { schema: null, rest: message };
  return { schema: m[1], domain: m[2], rest: message.slice(m[0].length) };
}
Plain English

Same regex idea as the backend's _TENANT_RE, in JavaScript. Captures schema and domain separately, and returns the message with the prefix stripped — so the Message column doesn't show the prefix twice (once in the Tenant column, once at the start of the message).

ReferenceGlossary

ContextVar
A Python primitive (from contextvars) for storing values that are scoped to the current execution context — typically a thread or async task. Used in stdout_proxy.py so per-request bindings don't bleed across threads.
django-tenants
The library Zango uses for multi-tenancy. Each tenant lives in its own Postgres schema; the active schema is set at the start of every request based on the domain. connection.tenant is the active tenant model.
ECS awslogs driver
The Docker log driver Amazon ECS uses to ship container stdout/stderr straight to CloudWatch Logs. Anything our containers write to stdout shows up in CloudWatch within seconds.
FilterLogEvents
The CloudWatch Logs API for querying log lines. Supports a "pattern" expression with limited operators — notably no OR over prefixes, which is why our multi-tenant filter does post-fetch filtering.
get_response
In Django middleware, the callable that runs the next middleware in the chain (or the view itself). You're sandwiched between the request coming in and the response going out.
handler / formatter / filter
Python logging primitives. A logger sends a record to its handlers. Each handler runs the record through its filters (to decide whether to emit at all) and its formatter (to produce a string). Handlers can be a file, stream, network socket, etc.
idempotent
Safe to call multiple times — the second and subsequent calls do nothing new. stdout_proxy.install() is idempotent so multiple boot paths can call it without doubling up.
loguru
A third-party Python logging library Zango uses for some logging in addition to stdlib logging. It has its own format string and sink system, independent of LOGGING.
propagate
A logger setting. When True (default), records bubble up to the parent logger's handlers after being processed by this logger's handlers. We set django.propagate = False so django records don't get processed twice.
react-router-dom
The library that maps URLs to React components. <Routes> contains <Route> entries; the first that matches wins. useParams and useNavigate are the hooks for reading params and navigating.
schema name
The Postgres schema for a tenant (e.g., newmicare, tntapp). Each tenant gets its own schema; queries run in that schema's namespace. Used as the tenant identifier throughout this PR.
task_prerun / task_postrun
Celery signals fired immediately before and after a task function executes. Reliable hook points for setting up and tearing down per-task state.
throttle (DRF)
Django REST Framework's rate-limiting mechanism. UserRateThrottle caps requests per user per time window — we cap browse at 30/min, tail at 60/min to protect CloudWatch quota.
wsgi.py / asgi.py
The standard entry points for Django apps running under a production server (gunicorn, uvicorn). manage.py runserver bypasses both — which is why we had to install the loguru sink from AppConfig.ready() as well.
