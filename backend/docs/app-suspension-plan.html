Zango Platform · design + implementation plan

App suspension — block traffic to suspended tenants, gracefully.

Platform admins should be able to suspend an app and instantly stop all traffic — HTTP requests, background tasks, worker jobs — while keeping the schema and data intact. Users who try to reach a suspended app see a calm, branded message telling them to contact support. This is a plan document: what to build, how the UX reads, and the exact code changes needed.

02 · Foundation

What already exists — and what doesn't

The TenantModel already carries a status field with "suspended" as one of the values, a suspended_on timestamp, and a suspend() method. What's missing is enforcement, the branded landing page, and the admin action.

Already in place

  • TenantModel.status — enum with Staged, Deployed, Suspended, Deleted
  • TenantModel.suspended_on — datetime, nullable
  • TenantModel.suspend() — sets status + timestamp, saves
  • Tenant resolution middleware (ZangoTenantMainMiddleware) that identifies the tenant per request
  • Platform apps list API + frontend page for admins

What's missing

  • Middleware check on tenant.status — currently traffic flows through regardless
  • The branded 404 template
  • An unsuspend() method (symmetric with suspend())
  • Platform admin API for suspend / unsuspend
  • Frontend suspend action, confirmation dialog, and status badge

04 · The suspended page

A calm, branded 404 — not a hostile error

HTTP 404 with a designed body. Amber accents, not red. Icon suggests "locked, but recoverable." Copy avoids blame. A single primary CTA to contact support keeps the page uncluttered. Data preservation is stated explicitly so anxious users don't panic.

https://myapp.zango.io/dashboard

Access to this application has been paused by the platform administrator. Your data is preserved and unchanged. Please contact support if you believe this is in error.

Reference · myapp.zango.io · 404
Design decisions

Amber not red: red reads as "system error" or "danger." Amber says "attention needed" without alarm. Serif italic on "suspended": the one word that carries the meaning gets typographic weight. "Your data is preserved": the most common panic — "did I lose my data?" — answered before the user asks. No CTA: a "contact support" button is tempting but the support channel is out-of-band (email, phone). The message tells users to contact support; the URL bar is where they go to another destination. Keeping the page action-free preserves calm and avoids stale mailto: links.

Suspending an app from Platform → Apps

Suspension lives inline in the existing apps list, not on a separate settings page. Admins see status at a glance via colored pills, and the action is one hover + one click + one confirmation away. 06 · Backend implementation

Middleware guard + branded template

Enforcement lives in ZangoTenantMainMiddleware.__call__, right after tenant resolution and before URL routing. A single conditional check on tenant.status short-circuits the request. The response is a rendered template returned with HTTP 404 so it's cacheable by CDNs and consistent with how other "resource unavailable" cases behave.

Exempt paths are load-bearing

Two exemptions matter: (a) health checks — if /api/v1/health/ returns 404, your load balancer marks the app as unhealthy and pages someone at 3am; (b) platform admin API — if /api/v1/platform/* returns 404, the operator can't unsuspend from the admin UI they just navigated to. Both exemptions must ship together with the middleware change.

Change 4 · Platform admin API endpoint

Single view for both actions. The action name lives in the URL — so logs, curl invocations, and audit trails all read at a glance without body inspection. Symmetric suspend / unsuspend collapse into one getattr(tenant, action)() call. Adding a future status action (archive, rotate-keys) is a one-line addition to the allowlist.

backend/src/zango/api/platform/apps/v1/views.py new 
class AppStatusActionView(ZangoGenericPlatformAPIView):
    """POST /api/v1/platform/apps/<app_uuid>/status/<action>/

    action ∈ {"suspend", "unsuspend"}.
    """
    permission_classes = (IsSuperAdminPlatformUser,)

    _ALLOWED_ACTIONS = ("suspend", "unsuspend")

    def post(self, request, app_uuid, action):
        if action not in self._ALLOWED_ACTIONS:
            return get_api_response(
                False,
                {"message": f"Unknown action '{action}'."},
                400,
            )
        tenant = get_object_or_404(TenantModel, uuid=app_uuid)
        getattr(tenant, action)()  # tenant.suspend() or tenant.unsuspend()
        return get_api_response(
            True,
            {"tenant": TenantSerializer(tenant).data},
            200,
        )

Why one view, not two

Read at a glance: \`POST /apps/<uuid>/status/suspend/\` reads as a command in logs, curl, and API docs. DRY body: the only difference between suspend and unsuspend is which method is called — getattr collapses that into a single line, no if/else duplication. Extensible: adding archive or rotate-keys in the future is one line in _ALLOWED_ACTIONS plus the matching method on the model. Auditable: the URL path is enough to distinguish suspend from unsuspend in access logs — no body parsing needed.

08 · Edge cases

What happens when…

Platform admin themselves visits the suspended app?

They still see the 404 page for the app's public URL. To interact with the tenant, they use the platform-admin API endpoints (which are exempt) — this includes the unsuspend action.

An API client with cached auth token calls a suspended app?

They get HTTP 404 with the branded HTML body. If the client only reads status codes, a 404 correctly signals "not available." If they parse HTML, they see the same page a browser sees.

A load balancer health check hits the app?

/api/v1/health/ is exempt from the suspend guard, returns 200 as normal. Load balancers don't mark suspended apps as unhealthy.

CDN caches the suspended page?

HTTP 404 is cacheable but short-TTL by default. If your CDN caches longer, add Cache-Control: max-age=60 to the response so unsuspend takes effect within a minute rather than hours.

09 · Celery + background work

Blocking traffic isn't enough — background work needs its own guard

HTTP middleware only sees HTTP requests. Celery tasks execute in worker processes and don't touch the middleware chain, so a middleware-only fix leaves several task sources unblocked: scheduled Celery Beat jobs, task chains that spawn from other tasks, workspace tasks (defined in workspaces/<tenant>/…/tasks.py), and codexec runs enqueued before the tenant was suspended. Each source has a different failure mode and a different fix.

The four sources — and what happens to each

A · HTTP-dispatched tasks

A view calls my_task.apply_async(...). Because the suspend guard returns 404 before the view body runs, the dispatch never happens. No additional work needed — middleware already covers this.

B · Celery Beat scheduled tasks

Zango has health_check_periodic_task, update_tool_usage_stats, and workspace beat schedules. These fire on a cron regardless of HTTP traffic. If the tenant is suspended, they'd still execute against the tenant's schema — writing data, sending emails, whatever the task does. Needs a task-entry guard.

C · Task chains + retries

A task that finishes with next_task.apply_async(...) spawns the next one from inside a worker. If the tenant is suspended between the parent and child, the child would still run. Same for retries scheduled with self.retry(countdown=...). Needs a task-entry guard.

D · In-flight tasks at the moment of suspension

A codexec run halfway through, or an email task that's mid-SMTP-send. Killing these mid-execution would leave DB rows in inconsistent states. Let them complete — no kill.

Framework-level enforcement — two chokepoints, no decorators

Requiring every task author to remember a decorator is fragile — someone will forget and the guarantee silently breaks. Instead, enforce the check in the two places every tenant-scoped task naturally passes through. Both are single-line additions in Zango core code. Task authors don't need to know this exists.

Chokepoint 1 · zango_task_executor

Every workspace task runs through zango.core.tasks.zango_task_executor — email/OTP/SMS sends, workspace-defined tasks, beat-scheduled workspace tasks, task chains, retries. The wrapper sets the tenant, then invokes the workspace-defined function via plugin_source.load_plugin. Adding the assertion right after connection.set_tenant(tenant) covers 100% of these.

Chokepoint 2 · codexec_executor

Codexec has its own task that sets the tenant and runs the user's snippet — it doesn't route through zango_task_executor. Add the same assertion here, immediately after set_tenant. One line.

The shared utility

backend/src/zango/apps/shared/tenancy/utils.py new 
from django.db import connection
from loguru import logger


class TenantSuspended(Exception):
    """Raised when a Celery task is invoked against a suspended tenant.
    Celery treats this as a task failure — the broker acknowledges the
    message and the failure is recorded. A future dispatch after
    unsuspend runs normally."""


def assert_tenant_active(tenant=None):
    """Raise TenantSuspended if the (given or current-connection) tenant
    is suspended. Cheap — one attribute lookup, no DB roundtrip."""
    tenant = tenant or getattr(connection, "tenant", None)
    if tenant is not None and tenant.status == "suspended":
        logger.info(
            "Task skipped: tenant '{}' is suspended",
            tenant.schema_name,
        )
        raise TenantSuspended(tenant.schema_name)

What we do not do

  • We don't use a task_prerun signal. Every tenant-scoped task in Zango passes through one of the two chokepoints above — the signal would be redundant, would need a platform-task skip-list to keep in sync with the actual task inventory, and would fire after Celery had already imported workspace code. The two explicit assertions are cheaper and clearer.
  • We don't purge the broker queue. Tasks enqueued before suspension stay in Redis. Each is caught on pickup by one of the two chokepoints — TenantSuspended propagates as a task failure, the broker acknowledges the message, no retry loop.
  • We don't kill in-flight tasks. Cancelling a mid-execution task with revoke(terminate=True) leaves the tenant's DB rows in whatever partial state the task had reached. Better to let a codexec run finish; the next dispatch is blocked.
  • We don't rely on per-task decorators. Task authors don't have to remember anything. The framework enforces the guard at the two natural chokepoints.
Why two chokepoints is enough

Every workspace task — email, OTP, SMS, workspace-defined, beat-scheduled, chained, retried — is routed to zango_task_executor by Zango's task registration machinery. The only exception is codexec_executor, which is a Zango core task that manages its own tenant context. Cover those two and there's nothing left to catch.
