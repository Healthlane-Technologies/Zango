Overview

What's being proposed

Two independent changes, each small in scope. They can ship together or separately. Both touch the backend only; the React UI requires no work.

Change 1

Use ZFileField for file columns

Replace models.FileField on CodeSnippetFile.file and CodeExecFile.file with Zango's ZFileField. Gets tenant-scoped storage paths, private S3 ACL, and Zango's standard upload convention for free.

Change 2

Block writes to settings.*

User snippets can still read settings.TIME_ZONE etc. (status quo), but any assignment to a settings.* attribute becomes a validator violation. About 15 lines added to validator.py.

Change 1 · ZFileField

Switch file columns to ZFileField

What ZFileField does

src/zango/core/storage_utils.py:64 defines a thin subclass of models.FileField that opinionates three things:

pyzango/core/storage_utils.py
class ZFileField(models.FileField): def __init__(self, ..., upload_to="", validators=[validate_file_extension], **kw): super().__init__( upload_to=(RandomUniqueFileName), # always · caller's value ignored validators=[validate_file_extension], # always · extension allowlist **kw, ) if make_public is False: self.storage.default_acl = "private"

The extension-allowlist issue

The only real friction. Two file roles to consider:

CodeSnippetFile · user-uploaded inputs

In the allowlist: .csv, .json, .xlsx, .pdf, .png, .zip.

NOT allowed: .txt, .yaml, .tsv, .parquet, .html, .md, .xml, .log.

CodeExecFile · snippet outputs

The user's snippet might do codexec.write("report.html", …) or write .parquet / .log output.

ZFileField would reject these at validation time. This is the harder problem.

Approach · ZFileField for both columns

Decision

Use ZFileField on both CodeSnippetFile.file and CodeExecFile.file

Symmetric field type across inputs and outputs. The extension allowlist is accepted as the contract — both human-uploaded inputs and snippet-written outputs must land in pdf · doc[x] · jpg · jpeg · png · svg · xls[x] · mp3 · wav · ppt[x] · zip · ico · mp4 · webm · csv · json.
Symmetric · no subclass · no special cases.
Tenant-scoped path + private ACL for free.
Consistent with how every other Zango model handles uploads.
Snippet outputs in unsupported extensions (.html, .parquet, .log, .txt) will raise ValidationError from file.save(). Documented as a constraint of the helper.

User-facing contract for the codexec helper

Document in the codexec helper docs that codexec.write(name, payload) requires name to end with one of the allowlisted extensions. Calls outside that set fail fast with a clear error rather than landing silently in storage.

pycodexec.py — error surface
# Inside _store_output, before obj.file.save(...) ext = os.path.splitext(name)[1].lower().lstrip(".") if ext not in ALLOWED_OUTPUT_EXTS: raise ValueError( f"codexec.write: extension '.{ext}' is not allowed. " f"Pick one of: {', '.join(sorted(ALLOWED_OUTPUT_EXTS))}", )

This turns the validator's deep ValidationError into a friendly error message at the public API boundary. The actual DB-level guard remains validate_file_extension.

Migration plan

Model changes

pyapps/code_execution/models.py — replace
# Before class CodeSnippetFile(FullAuditMixin): file = models.FileField(upload_to=_snippet_file_upload_to, max_length=512) class CodeExecFile(FullAuditMixin): file = models.FileField(upload_to=_exec_file_upload_to, max_length=512) # After from zango.core.storage_utils import ZFileField class CodeSnippetFile(FullAuditMixin): file = ZFileField(max_length=512) class CodeExecFile(FullAuditMixin): file = ZFileField(max_length=512)

Then remove the _snippet_file_upload_to and _exec_file_upload_to helpers — no longer used.

Generated migration

python manage.py makemigrations code_execution produces one AlterField per affected column. Django records it as a plain FileField (since ZFileField is a subclass), with the appropriate validator and upload_to attributes.

No data migration required

Django stores only the path string in the DB. The field-class change doesn't touch existing rows. Old files keep their old paths and continue to resolve — the storage backend just serves whatever string is in the DB. New files written after deploy land at the new layout.

Code points that need adjusting

Three places besides the model fields:

Location Change
models.py Drop _snippet_file_upload_to and _exec_file_upload_to helpers. Import ZFileField and swap both column types.
CodeSnippetRunView.post — snapshot-on-run No change. Still does file=snip_file.file.name to share the storage path; we're not triggering a new upload, just pointing the new row at the existing blob.
codexec.py _store_output Add an extension pre-check that raises a friendly ValueError before obj.file.save(...). The underlying file.save routes through RandomUniqueFileName and would otherwise raise ValidationError on a disallowed extension.

Storage layout · before and after

Before After
codexec/snippets/<snippet_uuid>/<file_uuid> <tenant>/CodeSnippetFile/<file_uuid>.<ext>
codexec/runs/<exec_uuid>/input/<file_uuid> <tenant>/CodeExecFile/<file_uuid>.<ext>

Note · we lose the "files for this run live under one prefix" grouping. Today an S3 ListObjectsV2 with prefix codexec/runs/<exec_id>/ shows only that run's files. After the switch, we'd query the DB instead. In practice we always do (via the CodeExecFile FK), so this is fine. Bulk GC would delete blob-by-blob instead of with a single prefix delete — marginally slower; not a real problem at our scale.

Filename preservation — unchanged

The user-facing filename continues to live on CodeSnippetFile.name / CodeExecFile.name. The download serializer's _signed_download_url generates S3 presigned URLs with ResponseContentDisposition set from that DB column. The actual storage key is the UUID-with-extension produced by RandomUniqueFileName — never seen by the user.

Change 2 · Validator hardening

Block writes to settings.*

User snippets should be able to read any django.conf.settings.X they want (status quo). Modifying a setting from a snippet must be rejected at save / pickup time. Today only a few specific reads (settings.SECRET_KEY, settings.DATABASES) are blocked; writes are not blocked at all.

Current state

The validator already blocks two specific attribute reads:

pyapps/code_execution/validator.py
BANNED_ATTR_PATHS = { ... ("settings", "SECRET_KEY"), ("settings", "DATABASES"), ... }

These prevent print(settings.SECRET_KEY) — fine to keep. But nothing blocks settings.TIME_ZONE = "UTC" or settings.DEBUG = True.

What changes

Extend the existing visit_Assign and visit_AugAssign guards to also fire when the assignment target's resolved chain starts with settings or matches django.conf.settings.….

Drop-in code

pyvalidator.py · inside visit_Assign, after the existing tenant-switch check
elif chain[0] == "settings" or chain[:3] == ("django", "conf", "settings"): self._add( node.lineno, node.col_offset, "DENY_SETTINGS_WRITE", f"Modifying '{'.'.join(chain)}' is not allowed; " f"settings are read-only.", ) # same idea inside visit_AugAssign for `settings.X += 1` etc.

Why the two-prong check works

The chain[:3] == ("django", "conf", "settings") check covers the import django.conf path where someone writes django.conf.settings.TIME_ZONE = "UTC".

The chain[0] == "settings" check covers the common path:

py
from django.conf import settings settings.TIME_ZONE = "UTC" # caught · chain = ("settings", "TIME_ZONE")

This works because visit_ImportFrom already records the alias map:

py
if alias.name in {"os", "shutil", "django", "settings", "connection"}: self.alias_to_module[local] = alias.name

So from django.conf import settings as cfg is also covered — cfg.X = Y becomes the chain ("cfg", "X"), gets resolved via the alias map to ("settings", "X"), and is matched.

Reflection bypasses — already covered

We don't need new checks for these — existing logic catches them:

Bypass attempt Already blocked by
setattr(settings, "X", value) setattr in BANNED_NAMES → DENY_NAME
delattr(settings, "X") delattr in BANNED_NAMES → DENY_NAME
settings.__dict__["X"] = value __dict__ in BANNED_DUNDERS → DENY_DUNDER
vars(settings)["X"] = value vars in BANNED_NAMES → DENY_NAME
settings.__class__.X = value __class__ in BANNED_DUNDERS → DENY_DUNDER
settings._wrapped.X = value Caught by the new check — chain starts with settings

Bonus catch · Django's LazySettings proxy

Django's settings is a LazySettings wrapper that proxies attribute access to an inner Settings instance stored on settings._wrapped. A determined attacker could try settings._wrapped.TIME_ZONE = "UTC". The chain is ("settings", "_wrapped", "TIME_ZONE") — the prefix-match on chain[0] == "settings" catches it. No extra logic needed.
