Zango platform · frontend search audit

Search is racing on the frontend and wrong on the backend.

Two independent classes of bug in the same feature. On the frontend, App Users shows results for the previous keystroke because two requests race and last-write-wins; Access Logs and Audit Logs debounce this away most of the time but still hit it under filter changes. On the backend, the queries themselves mis-treat integer IDs as text search targets, fire an extra 2 count() queries per keystroke on App Users, and re-run distinct+order_by on every keystroke against unindexed joins. Racing a slow, sometimes-wrong query is worse than either alone.

1 critical race · 3 correctness bugs 3 perf issues ~40 LOC frontend + ~30 LOC backend

All four are the same underlying issue: the useApi hook has no way to cancel a request in flight. When two requests are outstanding, whichever server response arrives last is displayed — regardless of which one the user actually asked for.

02 · Anatomy of the race

Why "the previous response overrides" happens

Real-world example — the user types "sm" into the App Users search, then continues to "smit". Both keystrokes trigger a fetch. The correct response for "smit" happens to be fast (rows are indexed well), but the "sm" query does a full-table scan on a slow join. Timeline:

t=0msusertypes "sm" → useEffect fires → GET /users?search=sm t=60msusertypes "smit" → useEffect fires → GET /users?search=smit t=90msserver200 OK for "smit" (12 rows) t=100msreactsetState(smit results) → UI shows the 12 rows ✓ t=340msserver200 OK for "sm" (413 rows) t=340msreactsetState(sm results) → UI overwrites with 413 rows ✗ t=340ms+usersees rows that don't match what the input shows

The user's input still says "smit" but the table now shows all 413 matches for "sm". Sometimes reloading fixes it. Sometimes typing another character fixes it. Sometimes it doesn't. This intermittence is what makes the bug hard to reproduce and easy to blame on caching or backend flakiness.

Why debounce helps but doesn't fix it

Debouncing collapses many keystrokes into one request. If the user types "smit" in 200 ms and debounce is 500 ms, only the "smit" request ever fires. Great. But debouncing doesn't help when the user changes a filter OR clears the search field OR pastes text — those actions frequently coincide with a still-in-flight request and produce the same overwrite. Also, if the user pauses briefly (500+ ms), types more, pauses again — two requests can be in flight legitimately.

Three problems compound:

  1. No debounce. Typing "smith" (5 keystrokes) fires 5 requests. Every character.
  2. No cancellation. The 4 obsolete requests continue to completion server-side and their responses still get processed client-side.
  3. No response versioning. The 5th response's arrival order isn't guaranteed to be last. If request #2 was slower (e.g. broader query), its late-arriving response calls updateAppUserManagementData and wipes out the correct #5 result.

05 · Fix strategy

One pattern, applied to all four surfaces

Two viable approaches. The one below wins on surgical footprint — it doesn't touch the shared useApi hook, it doesn't need a new abstraction, and it fixes each page independently.

Option A · Per-page request-ID versioning (recommended)

Track an incrementing counter in a useRef. Before starting a request, capture the counter's current value. After the request resolves, only apply the response if the counter still matches — meaning no newer request has been kicked off since.

Pattern — 6 lines added around each fetch Proposed 
const requestSeqRef = useRef(0);
const lastAppliedSeqRef = useRef(0);

const fetchUsers = async () => {
    const seq = ++requestSeqRef.current;   // bump + capture
    setLoading(true);

    const { response, success } = await triggerApi({ ... });

    // Discard stale responses: an older request landed after we
    // already applied a newer one, or a newer one is still coming.
    if (seq <= lastAppliedSeqRef.current) return;
    if (seq !== requestSeqRef.current) return;   // a newer one is in flight

    if (success && response) {
        updateAppUserManagementData(response);
        lastAppliedSeqRef.current = seq;
    }
    setLoading(false);
};

Cost: ~10 lines per page. Four pages. ~40 LOC total.

Wins: Fully local. Doesn't require touching the shared useApi hook. Works with the existing debounce (they compose — debounce reduces request count, versioning eliminates race). Doesn't leave server requests hanging (they complete server-side, we just ignore the response).

Option B · AbortController via useApi

Modify useApi to accept an AbortSignal, plumb it through callGetApi and friends, add per-page controllers. Cleaner conceptually, but three costs: (a) touches shared code that every caller uses, (b) requires each page to wire up controllers correctly, (c) server-side requests actually get cancelled which means the operator has to be careful about idempotency for POST/PUT.

Recommend saving Option B for a future refactor if we later want true request cancellation. Option A fixes the visible bug today.

Add debounce to App Users while we're there

App Users has no debounce today, so every keystroke fires a request. Even with versioning, that's wasteful and increases server load. Add a 300 ms debounce alongside the versioning fix.

Bonus — no more flicker

A nice side effect of versioning: when a stale response would have overwritten the good one, we now skip the setState entirely. No brief flicker of wrong data. The UI stays stable throughout the race window.

08 · Backend audit — is the search itself correct?

Access Logs + Audit Logs — search coverage + correctness

Beyond the race condition, the underlying get_queryset methods on the Access Logs, Audit Logs, and (spot-checked) App Users endpoints share several correctness and performance issues. Fixing the frontend race without fixing these leaves a fast UI on a slow, sometimes-wrong query.

Critical — wrong results

Performance
distinct() + order_by scan
3

Finding B1 — Int fields treated as text (Audit Logs)

Audit log free-text search silently misfilters when the term isn't numeric
backend/src/zango/api/platform/auditlogs/v1/views.py · lines 29–65 · AuditLogViewAPIV1.get_queryset
Critical

What happens. The mapping OR's every field into one Q():

views.py bug 
field_name_query_mapping = {
    "tenant_actor": "tenant_actor__name__icontains",
    "object_id": "object_id",            # int comparison — no icontains
    "id": "id",                          # same
    "object_repr": "object_repr__icontains",
    "changes": "changes__icontains",
    "object_uuid": "object_ref__object_uuid__icontains",
}
search_filters = {
    "id": self.process_id,
    "object_id": self.process_id,      # returns None on ValueError
}
for field_name, query in field_name_query_mapping.items():
    if search:
        if search_filters.get(field_name, None):
            filters |= Q(**{query: search_filters[field_name](search)})   # Q(id=None) when user typed "abc"
        else:
            filters |= Q(**{query: search})

When the user types "smith": process_id("smith") returns None, and Q(id=None) gets OR'd into the filter chain. Django treats this as WHERE id IS NULL — matches no rows (fine, because id is never null), but still gets planned and executed as part of the OR, adding cost to every keystroke.

The real bug. When the user types "12" (a plausible object_id), process_id("12") returns 12. The generated SQL includes OR id = 12 OR object_id = 12 OR ...text-fields LIKE '%12%'. The user probably meant one of those but gets all of them. That's semantically wrong for a free-text search — an object_repr containing "abc123" matches, an unrelated row with id=12 also matches, and there's no way to tell them apart in the result set.

Fix. Split integer fields into an explicit "advanced filter" panel (numeric input, one field at a time) OR drop them from the free-text OR chain and rely on the existing column-specific filters (?search_id=12) for numeric lookup.

Finding B2 — Same shape in Access Logs

Critical

Same pattern — id is int, user_id is int, both are OR'd into the text search chain. If the user types a numeric string like "42", they get an incorrect superset of rows.

Fix. Same as B1 — remove int columns from the OR chain, let column-specific filters handle them.

Finding B3 — App Users abuses id__icontains

Casting BigInt PK to text on every icontains match
backend/src/zango/api/platform/tenancy/v1/views.py · line 433
Performance

App Users maps "user_id" to "id__icontains". For every row, Postgres casts the BigInt PK to text before comparing:

generated SQL (approximate) bug 
WHERE CAST(id AS text) ILIKE '%42%'
   OR LOWER(email) LIKE '%42%'
   OR ...

The CAST … ILIKE can't use the BigInt PK index. On a workspace with 100k users, a single keystroke becomes a table scan (~100 ms in the best case). Multiply by keystrokes-per-second and the DB pool saturates.

Fix. Drop id__icontains. Either use exact id= match when the search string is numeric, or provide a separate "Filter by ID" input that only fires on exact match.

Finding B4 — .distinct() after OR of joined-field icontains

Distinct + order_by triggers full result-set sort per query
audit views.py:66 · access views.py:61 · users views.py:454
Performance

All three endpoints do records.filter(filters).distinct() after OR'ing icontains predicates that traverse FK joins (e.g. tenant_actor__name__icontains, roles__name__icontains). The join produces duplicate rows; distinct() deduplicates them. But combined with ORDER BY id DESC (or -modified_at) and offset-based pagination, Postgres must:

  1. Compute the full join
  2. Deduplicate the entire result
  3. Sort the entire deduplicated result
  4. Discard N-1 pages of rows to serve page N

Impact scales with table size. Under 10k rows: unnoticeable. At 100k+ rows: 300-800 ms per query. On a hot audit-log table: 2 seconds or more. And this fires on every keystroke (or every debounce interval).

Fix. Two options depending on how important the FK-name search is:

  • Move the FK-name search behind a separate "search by user" input that only kicks in when populated, and drop it from the free-text OR chain
  • Rewrite as a subquery: Q(tenant_actor__in=TenantActor.objects.filter(name__icontains=s)) so the join happens against a much smaller set

Either way, once the FK joins are out of the OR chain, .distinct() becomes unnecessary.

Finding B5 — App Users fires 3 queries per keystroke

Two extra .count() calls just for tile headers
backend/src/zango/api/platform/tenancy/v1/views.py · lines 465–466
Performance

The view fetches the paged users, then does filter(is_active=True).count() and filter(is_active=False).count() to compute the "Active users: N / Inactive users: M" tiles. That's three round trips per keystroke — one for the page + two for the tile counts.

Fix. Collapse into a single grouped query with Django's Count conditional aggregation:

proposed fix 
from django.db.models import Count, Q

counts = app_users_queryset.aggregate(
    active=Count("id", filter=Q(is_active=True)),
    inactive=Count("id", filter=Q(is_active=False)),
)

One query. Same numbers.

09 · Backend fix plan

The queries we should actually be running

Applied in the same PR as the frontend versioning fix so the two land together. All changes are internal to get_queryset methods; API contract unchanged.

B1 + B2 · Split numeric from text search

In both AuditLogs and AccessLogs, remove id, object_id, and user_id from the free-text field_name_query_mapping. If the operator wants an exact ID lookup, they use the column-specific search (?search_id=12) which is already supported.

B3 · Drop id__icontains from App Users

Replace with numeric-only branch: if search.isdigit(): filters |= Q(id=int(search)). Preserves capability without table-scanning on every keystroke.

B4 · Move FK-name searches to subqueries

Replace tenant_actor__name__icontains with tenant_actor__in=… against a small filtered subquery. Kills the need for .distinct() entirely.

B5 · Single aggregate for App Users tiles

Collapse the two .count() calls into one .aggregate(active=Count(...), inactive=Count(...)). Cuts DB round trips per keystroke from 3 to 1.

Indices worth adding (separate migration)

All three tables benefit from partial indices on the fields we search most. Audit logs already indexes timestamp; add functional indices on LOWER(object_repr) and LOWER(changes) for icontains prefix support. Access logs benefits from an index on (ip_address, attempt_time). This is a separate PR — measure the wins from B1-B5 first.

Appendix

Where the shared useApi hook falls short

frontend/src/hooks/useApi.jsx currently accepts {url, type, payload, loader, showErrorModal, notify}. It does not accept an AbortSignal, and it doesn't return an object with a .cancel() method. Its internal calls (callGetApi, callPostApi, etc.) similarly don't accept signals.

Adding cancellation is a straightforward refactor (~30 lines across useApi + the 4 service functions) but every caller becomes at-risk of a subtle change in error semantics — an aborted request returns {success: false, response: {message: 'aborted'}} today would need special handling. That's why Option A wins for this specific fix.
