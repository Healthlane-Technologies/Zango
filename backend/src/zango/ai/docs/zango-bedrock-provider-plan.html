Implementation Plan

AWS Bedrock LLM Provider — Zango AI Module

Author: Backend · Date: 2026-06-04 · Status: Draft

Context

The Zango AI module exposes an LLM provider abstraction (BaseLLMProvider) with three working providers — OpenAI, Anthropic, Azure OpenAI — plus a stub for AWS Bedrock at backend/src/zango/ai/providers/bedrock.py. The stub registers itself, defines config fields, and implements fetch_models() via boto3.bedrock.list_foundation_models, but complete() and stream() raise NotImplementedError.

This plan fills in the stub to make Bedrock a fully usable provider on parity with OpenAI and Anthropic. Two design decisions are locked:

  • API surface: Bedrock Converse API (bedrock-runtime.converse / converse_stream) — unified message + tool-use format across model families.
  • Auth: Static access keys only (existing config_fields unchanged).

No database migration is required — AppLLMProvider.provider_slug and config_encrypted are provider-agnostic, and the existing Fernet helpers in encryption.py already encrypt/mask any field marked type: "secret".

Overview

  • Primary file: backend/src/zango/ai/providers/bedrock.py — replace stub body, populate supported_models, implement complete() / stream(), override format_tools_for_api(), add a Bedrock-specific message converter.
  • New: backend/src/zango/tests/apps/ai/test_bedrock_provider.py — mock-based unit tests mirroring test_provider_client.py.
  • Packaging: add boto3 as an optional extra in backend/setup.py (do not promote to base install_requires).
  • Untouched: registry, base class, models, encryption, routing, migrations.

Notes on the Bedrock UX:

  • Two-secret pattern (Access Key + Secret Key) is the only structural difference from Anthropic / OpenAI — both render via the same generic config_fields renderer.
  • aws_region is a curated select, not free text — prevents typos and steers users to Bedrock-enabled regions.
  • The Default Model dropdown shows resolved IDs (with us. / eu. / apac. prefixes derived from the chosen region) — what you see is what's sent to AWS. The resolved for us-east-1 badge makes the derivation visible.
  • Changing the region re-resolves the model dropdown options — models whose region geography is unsupported (e.g. ca-central-1 + Claude 3.5 Sonnet v2) drop out of the list rather than showing a broken option.
  • "Test connection" hits validate_config() — which now also flags region/model geography mismatches up front, not as a runtime error.

Steps

  1. Step 1 — Populate supported_models with bare IDs + cross-region flag Replace the empty list with 6–10 entries covering Anthropic-on-Bedrock (anthropic.claude-3-5-sonnet-20241022-v2:0, anthropic.claude-3-5-haiku-20241022-v1:0, anthropic.claude-3-opus-20240229-v1:0), Amazon Nova (amazon.nova-pro-v1:0, amazon.nova-lite-v1:0, amazon.nova-micro-v1:0), Meta Llama (meta.llama3-1-70b-instruct-v1:0), and a Mistral entry. Each: id (bare, region-agnostic), name, context_window, max_output_tokens, input_cost_per_mtok, output_cost_per_mtok, supports_tools, supports_vision, supports_streaming, and a new requires_inference_profile: bool flag (True for Claude 3.5 Sonnet v2 and newer — they only work via cross-region profile; False for older Claude 3, Nova, Llama, Mistral). Pricing source: https://aws.amazon.com/bedrock/pricing/ (note URL + review cadence in a comment).
  2. Step 2 — Extend config_fields for parity + region dropdown Change aws_region from type: "string" to type: "select" with an inline options list of Bedrock-enabled regions (us-east-1, us-east-2, us-west-2, eu-west-1, eu-west-3, eu-central-1, ap-northeast-1, ap-south-1, ap-southeast-1, ap-southeast-2, ca-central-1, sa-east-1); default remains us-east-1. Append default_model (type: "select", options_from: "supported_models"), max_retries (default 2), and timeout_seconds (default 120). No separate "inference profile" field — the prefix is auto-derived from the chosen region (see Step 6). Frontend dependency: FE-2 in the Frontend Changes section must land alongside this step — the credential-field renderer in Providers.jsx needs a select branch or aws_region will silently degrade to a plain text input.
  3. Step 3 — Initialize the bedrock-runtime client in __init__ Lazy-import boto3 and botocore.config inside __init__. Build self._client = boto3.client("bedrock-runtime", aws_access_key_id=..., aws_secret_access_key=..., region_name=..., config=Config(read_timeout=timeout_seconds, retries={"max_attempts": max_retries})). Stash the ClientError / ReadTimeoutError classes on self for exception mapping. Note: unlike Anthropic/OpenAI SDKs, boto3 does not support per-call timeout overrides — the timeout is baked into the client at init. The agent-level timeout_seconds kwarg forwarded by client.py should therefore be used here at init time (read from config["timeout_seconds"]), making it consistent with how Anthropic reads it. Per-call override is not possible with botocore's current API.
  4. Step 4 — Implement _convert_messages_for_bedrock(messages) Walk list[LLMMessage] and produce Converse messages. Strings → {"text": s}. LLMFile images → {"image": {"format": "png|jpeg|gif|webp", "source": {"bytes": data}}}. Assistant tool calls → {"toolUse": {"toolUseId": id, "name": name, "input": dict}}. role="tool" entries → user message containing {"toolResult": {"toolUseId": tool_call_id, "content": [{"text": str_content}]}}. role="system" is stripped from messages and surfaced via the top-level system param.
  5. Step 5 — Override format_tools_for_api(tools) Return [{"toolSpec": {"name": t.name, "description": t.description, "inputSchema": {"json": t.input_schema}}} for t in tools]. Caller passes this as toolConfig={"tools": [...]}.
  6. Step 6 — Implement complete() with model-ID resolution Add a private helper _resolve_model_id(model): look the bare ID up in supported_models; if its requires_inference_profile is True, prepend the geography prefix derived from self.config["aws_region"] (us-* → us., eu-* → eu., ap-* → apac.; raise a clear LLMAPIError for unsupported geographies like ca-central-1 / sa-east-1). Pass-through if the model is not in the catalog (user override). Build kwargs (modelId=_resolve_model_id(model), messages, conditional system, inferenceConfig, conditional toolConfig). Time with time.monotonic(). Parse response["output"]["message"]["content"]: collect block["text"] entries; convert block["toolUse"] into LLMToolCall(id=toolUseId, name=name, input=input). Map response["stopReason"] via _map_stop_reason() (end_turn, tool_use, max_tokens, stop_sequence, content_filtered). Build LLMUsage(input_tokens=response["usage"]["inputTokens"], output_tokens=response["usage"]["outputTokens"]). Wrap exceptions: ClientError with code ThrottlingException → RateLimitExceeded; ReadTimeoutError → LLMTimeoutError; other ClientError → LLMAPIError(status_code=response["ResponseMetadata"]["HTTPStatusCode"], original_error=e). The timeout_seconds kwarg will be forwarded by client.py but should be silently ignored in complete()/stream() — boto3 does not support per-call timeouts, so the provider-level Config(read_timeout=...) set at __init__ is the effective timeout. Just consume the kwarg via kwargs.get("timeout_seconds") without using it to avoid unexpected-keyword errors.
  7. Step 7 — Implement stream() with converse_stream Resolve the model via _resolve_model_id() the same way as complete(). Iterate self._client.converse_stream(...)["stream"]. For contentBlockDelta: yield LLMStreamChunk(delta_text=...) for delta.text, or delta_tool_call={"partial_json": ...} for delta.toolUse.input. Capture stopReason from messageStop; capture usage from metadata. End with LLMStreamChunk(is_final=True, usage=..., stop_reason=_map_stop_reason(...)). Same exception mapping as complete().
  8. Step 8 — Tighten validate_config() with region/model compatibility check Drop the unused second client. Issue a single bedrock.list_foundation_models(maxResults=1) call to test credentials. Then, if the configured default_model has requires_inference_profile=True, verify the configured region maps to a supported geography (us-* / eu-* / ap-*) — otherwise return (False, "<model> requires a US, EU, or APAC region; <region> is not in a supported geography."). This catches misconfiguration up front rather than as a runtime ValidationException.
  9. Step 9 — Override get_models() to surface resolved IDs Return a copy of supported_models where each entry's id is the resolved ID for the configured region (i.e. us.anthropic.claude-3-5-sonnet-20241022-v2:0 when region is us-east-1). Entries whose requires_inference_profile=True but whose region has no geography mapping are filtered out (so the admin UI dropdown doesn't offer broken choices). The user sees exactly what will be sent to AWS — no hidden magic.
  10. Step 10 — Tests Create backend/src/zango/tests/apps/ai/test_bedrock_provider.py following the mock pattern in test_provider_client.py. Cover: complete() builds the correct converse() payload; tool calls extracted from toolUse blocks; stop-reason mapping; stream() yields text deltas then a final chunk with usage; ClientError("ThrottlingException") → RateLimitExceeded; missing boto3 surfaces as ValueError from fetch_models().
  11. Step 11 — Optional dependency In backend/setup.py, add extras_require["bedrock"] = ["boto3>=1.34"] (mirror the optional pattern used for openai / anthropic). Install via pip install zango[bedrock].

Frontend Changes

All AI-provider UI lives in one file: frontend/src/pages/appAi/components/Providers.jsx. Three separate things need to change there.

FE-1 — Remove Bedrock from COMING_SOON_PROVIDERS

Line 196 currently reads:

const COMING_SOON_PROVIDERS = ['bedrock', 'azure_openai'];

Remove 'bedrock' from the array:

const COMING_SOON_PROVIDERS = ['azure_openai'];

This is the only change needed to make the Bedrock card in the "Add Provider" wizard become selectable. The rest of the card behaviour (disabled state, opacity, non-clickable) is fully driven by membership in that array, so nothing else needs to change to un-gate it.

FE-2 — Add select field rendering to the credential-field loop

Both the Add Provider wizard (step 1 credential loop, ~line 499) and the Edit Provider form (credential loop, ~line 930) map over credentialFields and render only <input type="password"> or <input type="text">. There is no branch for field.type === 'select'.

The Bedrock backend plan changes aws_region to type: "select" with an inline options list. Because the credentialFields filter already passes through any select field whose options_from is not "supported_models", aws_region will appear in the list — but the current renderer falls through to a plain text input, losing the curated dropdown entirely.

Both loops must be extended with a select branch:

// In AddProviderWizard → credentialFields.map (step 1), ~line 499
// In EditProviderForm → credentialFields.map (~line 930)

if (field.type === 'select') {
  return (
    <div key={field.name}>
      <label className="font-lato text-[12px] font-semibold text-[#A3ABB1]">
        {field.label}{field.required && <span className="ml-[2px] text-[#EF4444]">*</span>}
      </label>
      <select
        value={credentials[field.name] ?? field.default ?? ''}
        onChange={(e) => setCredentials((prev) => ({ ...prev, [field.name]: e.target.value }))}
        className="mt-[4px] w-full rounded-[6px] border border-[#DDE2E5] px-[16px] py-[14px] font-lato text-[14px] text-[#111827] focus:outline-0 focus:border-[#5048ED] bg-white appearance-none cursor-pointer"
      >
        {(field.options || []).map((opt) => (
          <option key={opt.value} value={opt.value}>{opt.label}</option>
        ))}
      </select>
      {field.help_text && (
        <p className="mt-[3px] font-lato text-[11px] text-[#9CA3AF]">{field.help_text}</p>
      )}
    </div>
  );
}

The options shape expected by the frontend matches what the backend plan defines for aws_region: an array of { value, label } objects. The field.default fallback ensures us-east-1 is pre-selected without any user interaction.

The Edit form version differs only in the placeholder / keep-existing semantics: because region is not a secret, changing it takes effect immediately, so the normal onChange is correct. The "leave blank to keep existing" note in the Edit form credential section remains accurate for secret fields only — no copy change needed.

FE-3 — buildConfig must preserve the region default even if the user never touches it

The existing buildConfig() helper in AddProviderWizard (~line 328) already falls back to field.default when credentials[field.name] is empty:

} else if (field.default !== undefined) {
  config[field.name] = field.default;
}

This is correct — but only if the field is present in credentialFields. Since the filter for credentialFields already passes select fields that are not options_from: supported_models, and the new select branch pre-selects field.default into the dropdown, the initial credential state will be empty for aws_region (credentials is initialized to {}). Result: if the user leaves the dropdown at its default value without clicking, credentials["aws_region"] is undefined, and buildConfig correctly picks up field.default = "us-east-1". No change needed here — just document the invariant so no one "fixes" it incorrectly.

Risk Notes

⚠ Cross-region inference profile prefix is auto-derived from aws_region by _resolve_model_id(). Mapping covers us-*, eu-*, ap-*; ca-central-1 and sa-east-1 have no geography prefix, so models with requires_inference_profile=True are filtered from get_models() and rejected by validate_config() for those regions. Users wanting a different geography must change region — there is no separate inference-profile field by design.
⚠ Bedrock model availability varies by region. list_foundation_models already filters to the configured region, but users may be surprised if their region lacks Claude. Surface region clearly in the admin UI.
⚠ Not every Bedrock model supports Converse. Older Titan text and some Cohere variants don't. Curate supported_models to Converse-compatible entries and mark supports_streaming / supports_tools per model accurately.
⚠ boto3 is heavy (~30 MB with transitive deps). Keep it optional via extras_require; do not promote to base install_requires. The existing try/except ImportError in providers/__init__.py makes the absence non-fatal.
⚠ Bedrock pricing drifts. Hardcoded input_cost_per_mtok / output_cost_per_mtok will go stale. Add a source-URL comment and review on each release — mirroring how the Anthropic / OpenAI providers handle this.
⚠ boto3 has no per-call timeout override. Unlike the Anthropic and OpenAI SDKs (which accept timeout= on each .create() call), botocore bakes the timeout into the client at init via Config(read_timeout=...). The agent-level timeout_seconds field therefore only takes effect for Bedrock if it is read from config["timeout_seconds"] at provider init — not per call. Callers should be aware that changing an agent's timeout for a Bedrock-backed agent requires a provider reconfigure, not just an agent edit.
⚠ Streaming tool-call accumulation. Converse emits partial JSON for toolUse across multiple contentBlockDelta events. The stream loop must accumulate per contentBlockIndex and only finalize a tool call when the matching contentBlockStop arrives — otherwise downstream consumers see fragmented input.

Design Decisions

  • Region: aws_region is a curated dropdown of Bedrock-enabled regions; default us-east-1. Free-text invites silent typos — a dropdown is self-documenting and steers users away from non-Bedrock regions.
  • Model IDs: supported_models stores bare, region-agnostic IDs plus a requires_inference_profile flag. The geography prefix (us. / eu. / apac.) is derived from aws_region at call time — one catalog entry works for any region.
  • No separate inference-profile field: the prefix is fully determined by the already-selected region. A second field would either duplicate info or only ever have one valid option.
  • Resolved IDs are surfaced by get_models() — the admin UI dropdown shows exactly what will be sent to AWS (e.g. us.anthropic.claude-3-5-sonnet-20241022-v2:0), so there's no hidden magic.
  • Future: aws_session_token for STS / SSO is out of scope per the locked auth decision (static keys only).
