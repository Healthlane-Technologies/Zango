# Plan: Async Agent Run via User-Facing API

## Context

`AgentClient.run()` is fully synchronous — it blocks the request thread for the entire agentic loop (multi-round LLM calls + tool execution). For long-running agents this can take 30-120s+, making it unsuitable for direct HTTP calls.

The goal is to expose an async execution path: client fires a POST, gets back a `job_id` immediately (202), then polls for status/result. The Celery infrastructure already exists (`async_llm_complete` task, `ProviderClient.complete_async()`). We need to wrap the full `AgentClient.run()` in a task with DB-backed job state.

---

## What `tasks.py` currently does

| Task | Purpose |
|------|---------|
| `async_llm_complete` | Single raw LLM call via Celery (no agentic loop, no tools, no memory) |
| `update_tool_usage_stats` | Periodic aggregation of `AppLLMToolCall` stats onto `AppLLMTool` |

`async_llm_complete` is a low-level primitive — it's what `ProviderClient.complete_async()` uses for fire-and-forget single completions. It does **not** run the full agent loop (no tool rounds, no memory, no prompt rendering). We need a new `async_agent_run` task that wraps `AgentClient.run()`.

### Step 1 — `AppLLMAgentRun` model (`models/agent_run.py`)

```python
class AppLLMAgentRun(FullAuditMixin):
    STATUS_CHOICES = [
        ("pending",   "Pending"),
        ("running",   "Running"),
        ("completed", "Completed"),
        ("failed",    "Failed"),
        ("timeout",   "Timeout"),
    ]

    agent           = ForeignKey("ai.AppLLMAgent", on_delete=SET_NULL, null=True, related_name="runs")
    agent_name      = CharField(max_length=150)          # denormalized
    run_id          = UUIDField(db_index=True)            # same UUID used in AppLLMInvocation.run_id
    celery_task_id  = CharField(max_length=255, null=True, blank=True, db_index=True)
    status          = CharField(max_length=20, choices=STATUS_CHOICES, default="pending")
    triggered_by    = CharField(max_length=20, default="user")
    user_id_ref     = CharField(max_length=255, null=True, blank=True)

    # Input
    input_variables = JSONField(default=dict)
    input_messages  = JSONField(null=True, blank=True)   # if caller supplies raw messages
    session_id      = CharField(max_length=255, null=True, blank=True)

    # Result (populated on completion)
    result_content  = TextField(null=True, blank=True)
    result_usage    = JSONField(null=True, blank=True)   # {input_tokens, output_tokens, cost_usd}
    result_session_id = CharField(max_length=255, null=True, blank=True)
    total_rounds    = PositiveSmallIntegerField(default=0)

    # Error (populated on failure)
    error_message   = TextField(null=True, blank=True)
    error_type      = CharField(max_length=100, null=True, blank=True)

    # Timing
    started_at      = DateTimeField(null=True, blank=True)
    completed_at    = DateTimeField(null=True, blank=True)

    class Meta:
        ordering = ["-created_at"]
        indexes = [Index(fields=["agent", "-created_at"]), Index(fields=["run_id"])]
```

Key point: `run_id` is generated by the view at dispatch time and passed to both `AppLLMAgentRun` and the Celery task. The task passes it to `AgentClient.run(run_id=...)` so all `AppLLMInvocation` records share the same `run_id` — linking job record to audit log.

### Step 2 — `async_agent_run` Celery task (`ai/tasks.py`)

```python
@shared_task(bind=True, name="zango.ai.async_agent_run")
def async_agent_run(self, agent_run_pk, tenant_id, agent_id, variables, messages,
                    session_id, triggered_by, run_id_str):
    from django_tenants.utils import get_tenant_model
    from django.db import connection
    from zango.apps.ai.models.agent_run import AppLLMAgentRun
    from zango.ai.agent_client import AgentClient
    from zango.apps.ai.models.agent import AppLLMAgent
    from zango.apps.dynamic_models.workspace.base import Workspace
    from django.utils import timezone
    import uuid

    # 1. Set tenant context (critical for multi-tenancy)
    tenant = get_tenant_model().objects.get(id=tenant_id)
    connection.set_tenant(tenant)

    # 2. Update job to running
    run = AppLLMAgentRun.objects.get(pk=agent_run_pk)
    run.status = "running"
    run.started_at = timezone.now()
    run.celery_task_id = self.request.id
    run.save(update_fields=["status", "started_at", "celery_task_id"])

    try:
        with connection.cursor():
            ws = Workspace(connection.tenant, request=None, as_systemuser=True)
            ws.ready()

            agent = AppLLMAgent.objects.select_related(
                "provider", "system_prompt__active_version", "user_prompt__active_version"
            ).get(id=agent_id)

            llm_messages = None
            if messages:
                from zango.ai.providers.base import LLMMessage
                llm_messages = [LLMMessage(role=m["role"], content=m["content"]) for m in messages]

            agent_client = AgentClient(agent)
            response = agent_client.run(
                variables=variables or None,
                messages=llm_messages,
                session_id=session_id,
                triggered_by=triggered_by,
                run_id=uuid.UUID(run_id_str),
            )

        # 3. Update job to completed
        run.status = "completed"
        run.completed_at = timezone.now()
        run.result_content = response.content
        run.result_usage = {
            "input_tokens": response.usage.input_tokens,
            "output_tokens": response.usage.output_tokens,
            "cost_usd": str(response.cost_usd),
            "model": response.model,
            "latency_ms": response.latency_ms,
        }
        run.result_session_id = response.session_id
        run.save(update_fields=["status", "completed_at", "result_content",
                                "result_usage", "result_session_id"])

    except Exception as e:
        run.status = "failed"
        run.completed_at = timezone.now()
        run.error_message = str(e)
        run.error_type = type(e).__name__
        run.save(update_fields=["status", "completed_at", "error_message", "error_type"])
        raise
```

### Step 3 — Three new API views (`api/platform/ai/v1/views.py`)

**A. `AgentRunDispatchViewAPIV1`** — `POST /agents/<id>/run/`
- Validates agent exists and is enabled
- Generates `run_id = uuid.uuid4()`
- Creates `AppLLMAgentRun(status="pending", run_id=run_id, ...)`
- Dispatches `async_agent_run.delay(...)` passing the run PK, tenant_id, run_id
- Returns **202** with `{job_id, run_id, status: "pending"}`
- Reuses tenant setup pattern from `AgentTestViewAPIV1`

**B. `AgentRunStatusViewAPIV1`** — `GET /agents/<id>/runs/<job_id>/`
- Fetches `AppLLMAgentRun` by `job_id` (pk) scoped to `agent_id`
- Returns status + timing + invocation count (via `AppLLMInvocation.objects.filter(run_id=run.run_id).count()`)
- No result content in this endpoint — keeps polling lightweight

**C. `AgentRunResultViewAPIV1`** — `GET /agents/<id>/runs/<job_id>/result/`
- Returns full result when `status == "completed"` or error details when `status == "failed"`
- Returns 202 with `{status: "running"}` if not yet complete

## Design Decisions

1. **`run_id` generated in the view, not the task** — so it can be returned in the 202 response immediately. The caller can query `AppLLMInvocation.run_id` for detailed round-level audit logs.
2. **`AppLLMAgentRun` is a new model, not extending `AppLLMInvocation`** — invocations are immutable append-only audit logs; job state needs to mutate (pending → running → completed).
3. **No webhooks/WebSocket in MVP** — polling via status endpoint is sufficient. The two-endpoint design (lightweight status vs full result) keeps polling cheap.
4. **`AgentClient.run()` needs a `run_id` param** — currently `run_id = uuid.uuid4()` is generated unconditionally at line 322 of `agent_client.py`. Add `run_id: Optional[uuid.UUID] = None` to the signature and use `run_id = run_id or uuid.uuid4()`. This lets the Celery task pass the pre-generated UUID (returned in the 202 response) so `AppLLMInvocation` records link back to the job.

## Verification

1. Dispatch: `POST /agents/<id>/run/` → assert 202 + `job_id` returned immediately
2. Poll: `GET /agents/<id>/runs/<job_id>/` → watch status transition `pending → running → completed`
3. Result: `GET /agents/<id>/runs/<job_id>/result/` → assert `result_content` populated
4. Audit linkage: `AppLLMInvocation.objects.filter(run_id=run.run_id)` → assert invocation records exist
5. Failure path: disable provider → dispatch run → assert `status == "failed"` + `error_message` set
6. Tenant isolation: verify task sets correct schema before querying models
